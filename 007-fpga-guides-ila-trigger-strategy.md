## Trigger strategy в ILA: подробная заметка

**Trigger strategy** — это способ, которым ты определяешь **когда именно ILA должен начать (или завершить) захват данных**. В FPGA-отладке это критично: глубина буфера ограничена, и без грамотного триггера ты либо не поймаешь событие, либо получишь “шум” без полезной информации.

ILA поддерживает:

- **сравнение сигналов (comparators)**
- **логические условия (Boolean trigger equations)**
- **многоступенчатые триггеры (trigger sequences)**
- **storage qualification (условная запись в буфер)**  
    Эти механизмы позволяют строить достаточно сложные стратегии захвата, а не просто “записать всё подряд”.

---

# 1. Базовая идея trigger strategy

Trigger — это не просто “сигнал ошибки”. Это ответ на вопрос:

👉 **в какой момент времени мне нужно увидеть систему, чтобы понять причину сбоя?**

Хороший trigger должен:

- сработать **достаточно редко** (иначе буфер забьется мусором)
- сработать **достаточно точно** (чтобы попасть в нужный момент)
- дать **контекст до события** (pre-trigger)
- и **последствия после события** (post-trigger)

---

# 2. Основные режимы триггера

## 2.1 Простой уровень (level trigger)

Самый простой случай:

trigger = error == 1

Используется когда:

- ошибка редкая
- есть явный error flag

⚠️ Проблема:

- ты видишь момент ошибки, но не обязательно причину

---

## 2.2 Триггер по фронту (edge)

Например:

trigger = rising_edge(error)

Используется когда:

- сигнал может долго быть в 1
- важно поймать именно момент возникновения

---

## 2.3 Комбинационный trigger (Boolean)

Например:

trigger = (state == WRITE) && (ready == 0) && (timeout == 1)

Это один из самых полезных режимов:

- позволяет отсечь “ложные” срабатывания
- ловит именно нужную ситуацию

---

## 2.4 Trigger sequence (многоступенчатый)

ILA может ждать **последовательность событий**:

1. state == IDLE  
2. затем valid == 1  
3. затем error == 1

Используется когда:

- проблема возникает только после конкретной цепочки событий
- нужно поймать редкий сценарий

👉 Это один из самых мощных инструментов для сложных багов

---

## 2.5 Storage qualification (условная запись)

Это не совсем trigger, но часть стратегии:

👉 **в буфер записываются только те такты, где условие истинно**

Пример:

store only when valid == 1

Плюсы:

- сильно увеличивает “полезную глубину”
- убирает idle-шум

---

# 3. Pre-trigger vs Post-trigger

Очень важная часть стратегии — где находится trigger внутри буфера.

## Pre-trigger (до события)

- показывает **причину**
- обычно самое ценное

## Post-trigger (после события)

- показывает **последствия**
- важно для анализа реакции системы

👉 Типичный баланс:

- 60–80% pre-trigger
- 20–40% post-trigger

Если ошибка редкая — увеличивай pre-trigger  
Если важна реакция системы — увеличивай post-trigger

---

# 4. Выбор правильного trigger

## 4.1 Trigger должен быть “ближе к причине”

Плохо:

trigger = global_error

Хорошо:

trigger = (state == WAIT_DATA) && (timeout == 1)

👉 Чем ближе trigger к месту принятия решения — тем полезнее захват

---

## 4.2 Trigger должен быть редким

Если trigger срабатывает часто:

- буфер заполняется быстро
- нужное событие теряется

Решение:

- добавлять условия
- использовать sequence
- использовать qualifiers

---

## 4.3 Trigger должен быть устойчивым

Плохой trigger:

- на glitch
- на шумный сигнал
- на комбинаторику без регистрации

Лучше:

- использовать зарегистрированные сигналы
- использовать “стабильные” состояния (FSM, флаги)

---

# 5. Практические паттерны trigger strategy

## 5.1 Debug FSM

trigger = (state == ERROR)

- добавить:

- предыдущие состояния (pre-trigger)
- условия перехода

---

## 5.2 Timeout / зависание

trigger = timeout_flag

или лучше:

trigger = (state == WAIT) && (counter == MAX)

---

## 5.3 AXI / streaming

Плохой trigger:

trigger = valid

Хороший:

trigger = valid && !ready

👉 ловит backpressure

---

## 5.4 FIFO overflow

trigger = overflow

или:

trigger = (wr_en && full)

---

## 5.5 DDR / init

trigger = calib_fail

или:

trigger = (phase == READ_LEVELING) && (error == 1)

---

## 5.6 Редкий баг

Используй sequence:

1. start_transaction  
2. specific_condition  
3. error

---

# 6. Trigger и probe selection связаны

Trigger strategy напрямую зависит от того, какие probes ты выбрал.

Если у тебя нет:

- FSM state
- handshake
- counters

👉 ты не сможешь сделать хороший trigger

Поэтому:  
**probe selection и trigger strategy всегда проектируются вместе**

# 7. Как ловить очень редкие события

Это отдельный класс задач.

## Подход:

### 1. Добавить sticky flag

error_latched <= error_latched || error

### 2. Триггериться по нему

trigger = error_latched

### 3. Использовать storage qualification

store only when activity == 1

### 4. Увеличить pre-trigger

---

# 8. Частые ошибки

## ❌ Trigger слишком общий

→ много ложных срабатываний

## ❌ Trigger слишком узкий

→ событие никогда не ловится

## ❌ Нет pre-trigger

→ не видно причины

## ❌ Trigger на неправильном домене

→ событие не совпадает по времени

## ❌ Trigger на data без qualifier

→ ловится мусор

---

# 9. Практическая стратегия (пошагово)

### Шаг 1. Грубый trigger

- error / fail / timeout

### Шаг 2. Уточнение

- добавить state
- добавить handshake

### Шаг 3. Ужесточение

- добавить условия
- убрать лишние срабатывания

### Шаг 4. Sequence

- если баг сложный

---

# 10. Когда trigger вообще не нужен

Иногда лучше:

- включить **continuous capture**
- использовать storage qualification
- анализировать поток

Но это работает только если:

- проблема частая
- есть простой паттерн

---

# 11. Краткое резюме

**Trigger strategy** — это выбор условий, при которых ILA фиксирует интересующий момент времени.

Хороший trigger:

- редкий
- точный
- устойчивый
- близкий к причине
- дает хороший pre-trigger контекст

Основные инструменты:

- level / edge trigger
- Boolean conditions
- trigger sequences
- storage qualification

Главная инженерная идея:

👉 **Ты не просто ловишь ошибку — ты ловишь момент, который объясняет, почему она произошла.**
