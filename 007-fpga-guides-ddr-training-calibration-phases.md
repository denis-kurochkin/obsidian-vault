## Calibration phases в DDR-интерфейсе FPGA

Ниже — заметка именно про **фазы калибровки/training**, которые обычно проходят DDR3/DDR4 интерфейсы в FPGA-памяти PHY/контроллере. В реальном проекте это чаще всего делает готовый IP: у AMD/Xilinx — **Memory IP / MIG**, у Intel — **EMIF**. У AMD в последовательность калибровки входят, в частности, DQS gate, write leveling, read deskew/centering, write centering, sanity checks, multi-rank adjustments, VT tracking и, для некоторых конфигураций, Vref calibration. У Intel EMIF тоже используются фазы read calibration, write leveling, deskew/centering и отладка по calibration report.

### Зачем вообще нужны calibration phases

DDR-интерфейс работает на скоростях, где уже нельзя полагаться на “идеальные” фронты и одинаковые задержки. Разброс дают трассировка платы, различие длин линий, упаковка микросхемы, температура и питание. Поэтому после reset контроллер не просто “включает память”, а проходит набор фаз, в которых ищет правильные задержки для CK, DQS и DQ, чтобы попасть в центр допустимого окна данных.

---

## Общая логика последовательности

В практическом смысле calibration phases можно представить так:

1. **Подготовка PHY / базовая инициализация**
2. **Read gate / DQS gate calibration**
3. **Write leveling**
4. **Read leveling**
    - per-bit deskew
    - DQS centering
5. **Write calibration**
    - DQS-to-DQ
    - per-bit deskew
    - centering
6. **Проверки корректности**
7. **Коррекции для multi-rank**
8. **VT tracking / Vref-related stages**

Именно такая последовательность явно видена в AMD PG150: после инициализации идут DQS Gate, Write Leveling, Read DQS Deskew and Centering, Read Per-Bit Deskew, Read Sanity Check, Write DQS-to-DQ, Write DQS-to-DQ Per-Bit Deskew, Write DQS-to-DQ Centering, Write Latency Calibration, sanity checks, затем multi-rank adjustments, VT tracking и Vref calibration.

---

## 1) Базовая инициализация PHY и памяти

До собственно training контроллер должен:

- вывести DRAM из reset,
- настроить mode registers,
- поднять и стабилизировать тактирование,
- привести PHY в исходное состояние.

Практически это означает: до фаз centering/deskew у тебя уже есть работающий референсный clocking, базовая конфигурация DRAM и начальная установка внутренних delay-элементов PHY. У AMD перед read leveling отдельно упоминается BISC и то, что после него capture clock уже находится внутри valid window, но еще не в центре; дальше уже идет поиск краев окна и центрирование.

**Смысл фазы:** создать стартовую точку, от которой можно начинать измерять реальные временные окна.

---

## 2) DQS Gate / Read Gate calibration

Это одна из первых критичных read-фаз. Ее задача — определить, **когда вообще открывать окно приема** во FPGA, то есть в какой момент ожидать приход read burst. У AMD DQS gate calibration настраивает gate так, чтобы включение захвата произошло за один такт до первых валидных DQ; при этом регулируются coarse и fine tap’ы, а алгоритм ищет preamble DQS последовательно по байтам. Если preamble не найден, увеличивается read latency и поиск повторяется.

### Что физически делает эта фаза

При чтении DRAM не постоянно ведет DQS — он активен только во время burst. FPGA должна:

- поймать **preamble** DQS,
- открыть приемник вовремя,
- не открыть его слишком рано или слишком поздно.

### Если gate настроен плохо

- read data может не захватываться вообще;
- может захватываться не тот burst;
- дальнейшие read-фазы будут бессмысленны, потому что strobe найден неверно.

**Идея:** сначала научиться видеть сам burst, потом уже тонко настраивать место выборки внутри burst.

---

## 3) Write leveling

Это фаза выравнивания **write DQS относительно CK** на стороне DRAM. Она особенно важна для DDR3/DDR4 и fly-by topology, где clock/address/data приходят к микросхемам не одновременно. У AMD write leveling описан как процедура, где DRAM выдает feedback по DQ, а FPGA изменяет задержку DQS, пока не обнаружит переход 0→1; затем выполняется дополнительный шаг центрирования в шумовой области вокруг перехода для максимального запаса. Задержка DQS подбирается coarse и fine taps/ODELAY.

### Что именно подстраивается

- задержка **write DQS** для каждого byte lane / nibble;
- иногда отдельно coarse и fine часть задержки.

### Что означает write leveling по сути

Во время записи DRAM должна увидеть DQS в правильном положении относительно CK, чтобы принять данные корректно по спецификации tDQSS. Write leveling компенсирует перекос между forwarded clock и write strobes.

### Практический смысл для FPGA

После этой фазы write-путь уже не “примерно работает”, а привязан к реальной задержке трасс на плате и реальному положению clock у памяти.

---

## 4) Read leveling: per-bit deskew + DQS centering

После gate и write leveling следующая большая группа — **read leveling**. У AMD она делится на две фазы: **Per-Bit Deskew** и **Read DQS Centering**. Для DQS centering используются паттерны MPR DDR3/DDR4, а для per-bit deskew полезен более сложный, неповторяющийся шаблон, который сначала записывается в DRAM и потом считывается; в multi-rank системе этот шаг делается на первом ранге.

### 4.1 Per-bit deskew

У разных битов внутри одного byte lane задержка прихода может отличаться. Даже если общий DQS уже примерно на месте, отдельные DQ0…DQ7 могут приходить со смещением друг относительно друга.

**Задача deskew:**

- выровнять биты внутри группы данных;
- убрать внутригрупповой skew;
- сделать так, чтобы один и тот же sampling point годился для всех битов группы.

У Intel EMIF в обзоре калибровок также прямо фигурирует **deskew calibration**, причем read deskew выполняется до write leveling и минимум два раза — до write calibration и позже с более сложными шаблонами.

### 4.2 Read DQS centering

Когда биты внутри группы выровнены, нужно разместить read sample clock / capture point примерно в центре valid eye. У AMD read leveling описан как поиск левого и правого края valid window и последующее центрирование; делается это nibble-wise, потому что каждая nibble имеет собственный capture clock.

**Результат фазы:**

- FPGA знает, где левый край окна чтения;
- где правый край;
- и ставит точку выборки посередине, а не “где-то внутри”.

### Почему это отдельная фаза от gate calibration

- **Gate** отвечает за то, чтобы открыть окно приема в правильный burst.
- **Centering** отвечает за то, чтобы внутри этого burst снимать данные в центре глаза.

Это разные задачи, и на практике их нельзя надежно решить одним числом задержки.

---

## 5) Read sanity check

После базового read training обычно делается проверка, что найденные настройки реально дают правильные данные, а не просто “похожий” паттерн. У AMD после read deskew/centering явно присутствует **Read Sanity Check**.

### Смысл

- проверить, что training не сошелся на ложный локальный максимум;
- убедиться, что реальный read path устойчив;
- отловить ситуации, где простые паттерны проходят, а реальные данные уже нет.

Это уже не поиск коэффициентов, а валидация найденных коэффициентов.

---

## 6) Write calibration после write leveling

Write leveling решает только взаимное положение **write DQS и CK**. Но этого недостаточно: нужно еще убедиться, что сами **DQ корректно расположены относительно DQS** при записи. Поэтому у AMD дальше идут отдельные стадии:

- **Write DQS-to-DQ**
- **Write DQS-to-DQ Per-Bit Deskew**
- **Write DQS-to-DQ Centering**
- **Write DQS-to-DM/DBI**  
    

### Что это значит practically

После того как DQS выставлен относительно clock памяти:

- настраивается положение DQ относительно DQS;
- затем подравниваются отдельные биты;
- затем делается центрирование write window.

Иными словами, **write leveling — это не вся write calibration**, а только одна ее важная подфаза.

### Почему это важно

Можно идеально выровнять DQS по clock памяти, но все равно иметь ошибки записи, если отдельные DQ отстают/спешат относительно строба.

---

## 7) Write latency calibration

У AMD отдельно присутствует **Write Latency Calibration**. Это уже не только фазовые задержки на линиях, а согласование того, через сколько циклов после команды памяти/PHY нужно считать write-операцию приходящей в правильный момент относительно внутреннего datapath.

### Практический смысл

Нужно совместить:

- внутреннюю pipeline-латентность контроллера,
- задержки PHY,
- требования DRAM по таймингу команд/данных.

Это особенно важно, когда training делается не “на идеальной схеме”, а в реальном IP с глубокими внутренними pipeline.

---

## 8) Write/Read sanity checks

После write и read calibration обычно выполняются комбинированные проверки. У AMD это **Write/Read Sanity Check** и дальше еще complex centering stages.

### Зачем нужны complex stages

Простые паттерны удобны для поиска краев окна, но иногда не выявляют проблемы:

- межсимвольная интерференция,
- асимметрия по фронтам,
- зависимость от конкретного чередования битов,
- DBI/DM-специфические эффекты.

Поэтому в поздних стадиях могут использоваться **более сложные data patterns**, чтобы уточнить центрирование уже ближе к реальным условиям. У AMD Read DQ Training (Complex) прямо описан как финальная стадия подстройки read capture clock с использованием более сложных шаблонов, записываемых и считываемых из памяти.

---

## 9) Multi-rank adjustments

Если память многоранговая, одной калибровки “на интерфейс целиком” недостаточно. У AMD после основных фаз есть:

- **Read Leveling Multi-Rank Adjustment**
- **DQS Gate Multi-Rank Adjustment**
- **Write Latency Multi-Rank Check**
- **Write Read Sanity Check (Multi-Rank Only)**  
    

### Почему это нужно

У разных рангов:

- разные физические пути,
- разные нагрузки,
- разные задержки прихода/возврата сигнала.

Значит, коэффициенты, найденные для rank 0, не обязаны быть оптимальными для rank 1.

### Практический вывод

В multi-rank системе training почти всегда хранит и применяет **отдельные значения задержек по рангам**.

---

## 10) VT tracking

После initial calibration температура и напряжение продолжают жить своей жизнью. У AMD в последовательности явно присутствует **Enable VT Tracking**, то есть дальнейшее отслеживание вариаций voltage/temperature.

### Что это означает

Даже если training прошел идеально при старте:

- нагрев FPGA,
- изменение питания,
- дрейф DLL/PLL/IO delay

могут сдвинуть рабочую точку. VT tracking позволяет компенсировать часть такого дрейфа без полного повторного старта интерфейса.

### Важно

Это не всегда полноценный retraining всех фаз; чаще это сопровождение уже найденной рабочей точки.

---

## 11) Vref calibration

В AMD sequence фигурирует **Read and Write VREF Calibration**. Для DDR4 и новее это может быть важной частью настройки, потому что положение “глаза” определяется не только по времени, но и по уровню.

### Смысл

Даже при идеальном timing можно иметь плохой запас по уровню сигнала. Подбор reference voltage помогает:

- увеличить margin по входу,
- лучше центрировать eye по вертикали,
- повысить устойчивость на высоких скоростях.

### Когда особенно важно

- высокие частоты,
- плотная разводка,
- пограничный SI,
- сложные топологии или тяжелая нагрузка.

---

## Как все эти фазы связаны между собой

Хорошая ментальная модель такая:

- **DQS gate** отвечает: _“когда начинается burst чтения?”_
- **Write leveling** отвечает: _“где должен стоять write DQS относительно CK у DRAM?”_
- **Read deskew** отвечает: _“как выровнять отдельные биты DQ?”_
- **Read centering** отвечает: _“где центр read eye?”_
- **Write deskew/centering** отвечает: _“где центр write eye?”_
- **Sanity/complex stages** отвечают: _“точно ли найденные параметры устойчивы на реальных данных?”_
- **Multi-rank adjustment** отвечает: _“как скорректировать это для каждого ранга отдельно?”_
- **VT/Vref** отвечают: _“как удержать margin после старта?”_

Это не набор случайных шагов, а движение от грубой синхронизации к тонкой оптимизации окна.

---

## Что обычно видит разработчик FPGA

В реальном проекте ты чаще всего не пишешь эти алгоритмы сам, а работаешь с результатами IP:

- флаг **calibration done / init_calib_complete**;
- коды статусов и ошибок;
- debug taps / delay values;
- отчеты калибровки;
- ILA/VIO debug.

AMD прямо указывает, что debug interface содержит calibration status и tap settings, которые можно читать во время работы через Hardware Manager, Tcl или Memory IP Debug GUI.

### На практике это означает

Если training не проходит, смотришь не “память сломана или нет”, а **на какой фазе он упал**:

- не найден DQS preamble,
- не прошел write leveling,
- слишком узкое read window,
- не сошелся per-bit deskew,
- проблема только на rank 1,
- margin исчезает при нагреве.

---

## Что чаще всего ломает calibration

Хотя сами документы в приведенных фрагментах здесь не перечисляют полный список причин, из структуры training напрямую следует, что проблемы обычно связаны с тем, что calibration не может найти стабильные окна или края окна оказываются слишком близко. На практике это происходит из-за:

- ошибок разводки и несогласованных длин,
- плохого clocking,
- неверной конфигурации памяти/IP,
- слишком малого SI margin,
- проблем с питанием/Vref,
- неверного pin planning.  
    Общая важность board design и routing guidelines явно подчеркивается и в AMD, и в Intel документации по Memory IP/EMIF.

---

## Краткое резюме

**Calibration phases** в DDR FPGA — это поэтапная настройка интерфейса от грубого захвата burst до центрирования реального data eye.  
Типовая логика такая:

- поймать момент прихода чтения (**DQS gate**),
- выровнять write strobe относительно clock у DRAM (**write leveling**),
- выровнять биты чтения и поставить read sample в центр глаза (**read deskew + read centering**),
- выровнять write DQ относительно DQS (**write deskew + write centering**),
- проверить найденные параметры на реальных шаблонах (**sanity / complex training**),
- скорректировать их для rank’ов и условий среды (**multi-rank, VT, Vref**).
