**Digital Design / RTL Coding discipline** — это набор правил, по которым пишут RTL так, чтобы схема была **предсказуемой, синтезируемой, читаемой и безопасной для сопровождения**.

Если совсем по-простому, идея такая:

- RTL пишут не “чтобы симуляция прошла”, а **чтобы железо получилось корректным**.
    
- Код должен ясно показывать:
    
    - где **комбинаторика**,
        
    - где **регистры**,
        
    - где **границы тактовых доменов**,
        
    - где **reset**,
        
    - где **валидный протокол обмена**.
        
- Хорошая discipline уменьшает:
    
    - случайные latch,
        
    - гонки,
        
    - CDC-ошибки,
        
    - неоднозначное поведение между sim и synth,
        
    - сложность отладки.
        

Обычно в это понятие входят такие принципы:

**1. Четкое разделение combinational и sequential logic**  
Например:

- `always_comb` / `assign` — только для комбинаторики;
    
- `always_ff @(posedge clk)` — только для регистров.
    

**2. Неблокирующие и блокирующие присваивания использовать осознанно**  
Классическое правило:

- в sequential логике — обычно `<=`
    
- в combinational — обычно `=`
    

**3. Не создавать latch случайно**  
Если комбинаторный блок описан не полностью, синтезатор может сделать latch, хотя ты этого не хотел.

**4. Один регистр — одна понятная ответственность**  
Не перегружать один always-блок десятком несвязанных функций, если это ухудшает читаемость.

**5. Явная работа с reset**  
Нужно понимать:

- reset синхронный или асинхронный;
    
- где он действительно нужен;
    
- какие регистры обязаны инициализироваться.
    

**6. Дисциплина clock domain crossing**  
Нельзя просто так перекидывать сигнал между разными clock domain.  
Нужны synchronizer, FIFO, handshake, pulse transfer и т.д.

**7. Детерминированные FSM**  
Для автоматов важно:

- понятное разделение `state` / `next_state`;
    
- полное покрытие переходов;
    
- безопасное поведение по умолчанию.
    

**8. Понятные интерфейсы и протоколы**  
Например, у модуля должно быть ясно:

- когда вход valid,
    
- когда данные принимаются,
    
- когда выход считается достоверным.
    

**9. Параметризация без излишней магии**  
Параметры полезны, но если их слишком много и они связаны неочевидно, RTL становится хрупким.

**10. Код пишется для команды, а не только для синтезатора**  
Имена сигналов, структура модулей, комментарии, единый стиль — это тоже часть discipline.

Итоговая мысль такая:

> **RTL Coding discipline — это инженерная аккуратность в описании железа.**  
> Она нужна не для “красоты кода”, а чтобы проект стабильно переживал синтез, timing closure, отладку и передачу другому разработчику.
