**RTL flow control** — это способ согласовать передачу данных между блоками, когда один блок **производит** данные, а другой **принимает**, и их скорости не всегда совпадают.  
Главная задача — не потерять данные, не принять мусор и не сломать тайминг/конвейер.

Самая частая схема — **valid/ready**.

## Базовая идея valid/ready

Есть источник и приемник.

- **valid** — источник говорит: _“на выходе сейчас лежат корректные данные”_
- **ready** — приемник говорит: _“я могу принять данные в этом такте”_

Передача происходит только в тот такт, когда одновременно:

```
valid=1, ready=1
```

То есть:

- источник выставляет `data` и `valid`
- приемник выставляет `ready`
- если оба сигнала активны на фронте такта, слово считается переданным

Это очень удобная схема для FPGA, потому что она:

- хорошо ложится на pipeline
- естественно поддерживает stalls
- позволяет строить модульные интерфейсы
- используется в AXI-Stream и во многих внутренних datapath

---

## Что такое backpressure

**Backpressure** — это ситуация, когда downstream-блок не готов принимать данные и через `ready=0` тормозит upstream.

Пример:

- генератор данных может выдавать слово каждый такт
- следующий блок временно занят
- он опускает `ready`
- источник обязан удерживать `valid` и `data`, пока приемник не станет готов

То есть backpressure — это **распространение остановки вверх по потоку**.

Это один из ключевых механизмов управления потоком в FPGA-конвейерах.

---

## Почему это важно в FPGA

В FPGA почти всегда есть неидеальное совпадение скоростей между блоками:

- один блок считает каждый такт, другой — раз в несколько тактов
- память/DDR дает данные с паузами
- downstream может зависеть от арбитража
- разные стадии pipeline могут иметь переменную latency
- FIFO может быть почти полон или пуст

Без flow control возникают типовые проблемы:

- потеря данных
- дублирование данных
- чтение невалидного слова
- переполнение буфера
- underrun
- сложные неустойчивые state machine

---

## Типовые правила valid/ready

### Для источника

Источник должен:

- выставлять `valid=1`, когда данные готовы
- удерживать `data` стабильным, пока передача не произошла
- не снимать `valid`, пока `ready=0`, если именно это слово еще не принято

### Для приемника

Приемник должен:

- выставлять `ready=1`, когда может принять данные
- принимать слово только при `valid && ready`
- не считать данные принятыми, если `ready=0`

Это фундаментальная дисциплина интерфейса. Если обе стороны соблюдают ее одинаково, система ведет себя предсказуемо.

---

## Где это используется

В FPGA flow control встречается почти везде:

- между pipeline stage
- в stream processing
- в AXI-Stream
- между DMA и обработкой
- на границе с FIFO
- в memory read/write path
- в packet processing
- в видео и DSP-конвейерах

---

## Что важно понимать архитектурно

### 1. valid/ready — это не просто “два служебных сигнала”

Это модель контракта между блоками.

### 2. backpressure может распространяться далеко

Если один блок тормозит, остановка может пройти через несколько upstream-стадий.

### 3. длинняя цепочка `ready` может стать критическим comb path

Это очень частая FPGA-проблема. Иногда именно логика backpressure ограничивает Fmax, а не datapath.

### 4. для разрыва давления часто нужны FIFO/skid buffer/register slice

Они улучшают тайминг и делают поток устойчивее.

### 5. variable-latency блоки почти всегда требуют явного flow control

Без него их трудно безопасно вставить в pipeline.

---

## Типовые варианты поведения pipeline

### Fully streaming

Каждая стадия обычно готова, `ready=1` почти всегда, данные идут каждый такт.  
Лучший случай для throughput.

### Stallable pipeline

Иногда одна стадия опускает `ready`, и весь поток останавливается.  
Нормальный и очень частый режим.

### Buffered pipeline

Между стадиями стоят FIFO или skid buffers, поэтому кратковременные stalls не сразу останавливают весь тракт.

---

## Практический смысл backpressure

Backpressure полезен, потому что позволяет строить **lossless** поток без жесткого глобального расписания.  
Но у него есть цена:

- сложнее логика
- хуже timing на длинных ready-цепях
- появляются corner cases при stall/restart
- надо внимательно проверять handshake
