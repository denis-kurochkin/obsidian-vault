В RTL **pipeline** — это способ разрезать длинный путь обработки данных на несколько стадий, разделённых регистрами. Главная идея простая: вместо того чтобы пытаться сделать всю логику за один такт, мы делаем часть работы, сохраняем промежуточный результат в регистр, а потом продолжаем на следующем такте.

## Что такое pipeline в цифровом дизайне

Без pipeline у тебя есть схема вида:

`input -> combinational logic -> output`

Если комбинаторная логика слишком длинная, то период такта должен быть большим, иначе данные не успеют распространиться от регистра к регистру.

С pipeline схема становится такой:

`reg -> logic1 -> reg -> logic2 -> reg -> logic3 -> reg`

Теперь каждая стадия делает меньше работы за такт, и тактовую частоту можно поднять.

## Главная польза pipeline

Pipeline нужен в первую очередь для **роста частоты** и **улучшения timing**.

Что это даёт:

- меньше критический путь;
- выше Fmax;
- проще закрывать timing;
- можно строить более сложную обработку данных.

Но есть цена:

- растёт **latency** — результат появляется не сразу, а через несколько тактов;
- увеличивается число регистров;
- сложнее управлять valid/ready, flush, stall, reset;
- появляется риск ошибок выравнивания данных и служебных сигналов.

## Throughput и latency

Это ключевая пара понятий.

**Latency** — сколько тактов проходит от входа данных до выхода результата.  
**Throughput** — как часто можно выдавать новый результат.

Пример:

- 1 стадия без pipeline: результат через 1 такт, но частота низкая.
- 4 стадии pipeline: результат через 4 такта, зато после заполнения pipeline можно выдавать по одному результату каждый такт.

То есть pipeline почти всегда делают ради **throughput/Fmax**, а не ради уменьшения задержки.

## Базовый принцип RTL pipeline

Каждая стадия обычно состоит из:

- набора входных регистров;
- комбинаторной логики стадии;
- выходных регистров.

Простейший пример:

```verilog
always @(posedge clk) begin
    stage1 <= a + b;
    stage2 <= stage1 * c;
    y      <= stage2 ^ d;
end
```

Здесь:

- `stage1` — результат первой стадии,
- `stage2` — второй,
- `y` — итог после третьей.

Физически это означает, что данные идут по цепочке стадий, по одной стадии за такт.

## Как выбирать границы стадий

Хороший pipeline стараются делить так, чтобы:

- задержка каждой стадии была примерно одинаковой;
- между регистрами не было слишком длинной логики;
- не было стадии “почти пустой”, а рядом — очень тяжёлой;
- данные и управляющие сигналы проходили одинаковое число стадий.

Типичная ошибка: запайплайнили данные, но забыли запайплайнить `valid`, `last`, `id`, `addr`, `keep` или другие служебные сигналы. Тогда на выходе всё “съезжает”.

## Что обычно пайплайнят

Часто pipeline применяют в:

- арифметике: sum, multiply, MAC, CRC, FIR, FFT;
- трактах обработки данных;
- адресных и командных путях;
- интерфейсах AXI/AXIS/Avalon;
- декодерах/энкодерах;
- FSM с тяжёлой логикой next-state/output.

## Виды pipeline

### Data pipeline

Обычный поток данных по стадиям.  
Пример: входное слово → преобразование → упаковка → выход.

### Control pipeline

Пайплайнятся не только данные, но и признаки:  
`valid`, `last`, `sof`, `eof`, `id`, `user`.

### Arithmetic pipeline

Разбиение сложной математики на стадии. Часто у DSP-блоков внутри уже есть встроенные pipeline-регистры.

## Критический путь и зачем регистры помогают

Timing-path обычно выглядит так:

`launch flop -> combinational logic -> capture flop`

Если логика слишком большая, нарушается setup.  
Когда добавляешь промежуточный регистр, один длинный путь превращается в два более коротких.

По сути pipeline — это сознательное управление длиной combinational path.

## Balance pipeline stages

Одна из главных практических тем: **балансировка**.

Например, если у тебя:

- стадия 1 = 1 ns
- стадия 2 = 7 ns
- стадия 3 = 1 ns

то pipeline сделан плохо, потому что Fmax всё равно ограничит стадия 2.  
Нужно стремиться к более равномерному делению.

## Valid pipeline

В реальных схемах мало просто передавать данные. Нужно знать, какие данные “настоящие”.

Обычно вместе с данными идёт `valid`:

always @(posedge clk) begin  
    if (rst) begin  
        v1 <= 1'b0;  
        v2 <= 1'b0;  
    end else begin  
        if (in_valid) begin  
            d1 <= in_data + 1;  
        end  
        v1 <= in_valid;  
  
        if (v1) begin  
            d2 <= d1 ^ 8'h5A;  
        end  
        v2 <= v1;  
    end  
end

Здесь `v1`, `v2` идут рядом с данными и показывают, какая стадия содержит валидный результат.

## Stall и bubble

В более сложных pipeline бывают:

- **stall** — остановка стадии, когда следующая стадия не готова;
- **bubble** — пустой такт/пустая запись в pipeline;
- **flush** — сброс содержимого pipeline, например после ошибки или смены контекста.

Это уже важная тема для потоковых интерфейсов и CPU-подобных трактов.

## Pipeline и ready/valid

Если говорить про RTL и AXI-stream стиль, очень важно понимать, что pipeline может быть:

- просто регистровый сдвиг без backpressure;
- pipeline с `valid`;
- pipeline с `ready/valid`, где возможна остановка потока.

Во втором случае уже нужно аккуратно проектировать, чтобы при `ready=0` данные не терялись и не дублировались.

## Частые ошибки

Самые типичные:

1. **Сигналы не выровнены по стадиям**  
    Данные прошли 3 стадии, а `valid` только 2.
2. **Использование blocking assignment в sequential логике**  
    Для регистровых стадий обычно нужны `<=`.
3. **Слишком длинная combinational логика внутри одной стадии**  
    Формально pipeline есть, practically timing всё равно плохой.
4. **Непродуманный reset**  
    После reset pipeline может выдавать мусор, если не сброшены valid-флаги.
5. **Лишний pipeline там, где он не нужен**  
    Можно легко усложнить схему без реальной пользы.
6. **Нарушение handshaking**  
    Особенно в ready/valid pipeline.

## Когда pipeline не нужен

Не всегда его стоит делать. Например:

- логика и так короткая;
- важнее минимальная latency, а не частота;
- ресурс регистров критичен;
- интерфейс редкий и timing закрывается без pipeline.

То есть pipeline — это инструмент, а не правило “всегда делай больше стадий”.

## Практический образ мышления

Когда смотришь на RTL-блок, полезно задавать себе вопросы:

- Где здесь критический путь?
- Можно ли разделить логику на 2–3 естественные стадии?
- Какие сигналы обязаны идти вместе с данными?
- Что будет при reset?
- Нужны ли stall/flush?
- Что важнее: latency или Fmax?
- Нет ли уже встроенного pipeline в DSP/BRAM/IP?
