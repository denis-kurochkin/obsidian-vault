# RTL / Digital Design No Unintended Latches

## Что это такое

**No unintended latches** — это принцип RTL coding discipline, который означает:

> в дизайне не должно появляться latch, если ты не создавал их намеренно и осознанно.

Проще говоря, если ты пишешь combinational logic, синтезатор не должен вдруг решить, что для сохранения значения нужен **level-sensitive latch**, только потому что описание логики получилось неполным.

В FPGA и обычном RTL-проектировании случайный latch чаще всего считается:

- ошибкой кодирования;
    
- признаком неполного combinational description;
    
- источником трудноуловимых проблем.

Именно поэтому принцип формулируют не как "никогда не использовать latch вообще", а как:

**не допускать непреднамеренных latch.**

---

## Что такое latch простыми словами

Latch — это элемент памяти, который:

- зависит от уровня управляющего сигнала, а не от фронта clock;
    
- может "прозрачно" пропускать значение, пока enable активен;
    
- удерживает предыдущее значение, когда enable неактивен.

Это отличается от flip-flop, который обновляется по фронту clock.

В RTL-проектах для FPGA основной базовый стиль почти всегда строится вокруг:

- combinational logic;
    
- edge-triggered registers.

Поэтому latch в таком дизайне обычно является **неожиданным гостем**, а не желаемым элементом архитектуры.

---

## Почему unintended latch — это проблема

Случайный latch неприятен сразу по нескольким причинам.

### 1. Он появляется не по архитектурному замыслу

Ты хотел чистую combinational logic, а получил память.

### 2. Поведение схемы становится менее очевидным

Вместо "выход — это функция текущих входов" получаем:

- иногда новое значение,
    
- иногда удержание старого.

### 3. Сложнее timing и analysis

Latch имеет другую временную природу, чем обычные регистры.

### 4. Сложнее debug

В симуляции и на waveform поведение может выглядеть странно: сигнал как будто "запоминает" старое значение без явного регистра.

### 5. Часто это симптом неполного описания логики

То есть latch — не корень проблемы, а индикатор того, что combinational block написан не полностью.

---

## Главная идея

Если ты пишешь **combinational block**, то для любого набора входов выход должен быть определен **всегда**.

Если для каких-то условий новое значение не назначено, синтезатор делает вывод:

> "раз значение должно сохраняться, значит нужна память"

И этой памятью часто становится latch.

То есть unintended latch почти всегда рождается из логики вида:

- не все ветки `if/else` покрыты;
    
- не все варианты `case` описаны;
    
- не всем выходам присваивается значение во всех путях.

---

## Самая типичная причина latch

## Неполное присваивание в combinational logic

Например:

```verilog
always @(*) begin
    if (en)
        y = a;
end
```

Что здесь происходит, когда `en = 0`?

Для `y` новое значение не задано. Значит его надо удержать. Чтобы удержать значение, нужна память. Синтезатор может создать latch.

То есть проблема не в `if` как таковом, а в том, что **нет полного описания выхода `y`**.

---

## Правильная логика мышления

Combinational block должен отвечать на вопрос:

> чему равен каждый выход **всегда**, а не только в удобных ветках?

Если ты не можешь однозначно ответить, что будет с выходом в каждом случае, риск latch высокий.

---

## Типовые ситуации, в которых появляются unintended latch

## 1. `if` без `else`

Пример:

```verilog
always @(*) begin
    if (sel)
        out = in1;
end
```

Когда `sel = 0`, `out` не назначен. Значит нужен latch.

---

## 2. Неполный `case`

Пример:

```verilog
always @(*) begin
    case (state)
        2'b00: out = a;
        2'b01: out = b;
    endcase
end
```

Что делать при `state = 2'b10` или `2'b11`? Значение `out` не определено → возможен latch.

---

## 3. Не всем сигналам присваивается значение во всех ветках

Пример:

```verilog
always @(*) begin
    if (mode) begin
        a_out = a;
        b_out = b;
    end else begin
        a_out = c;
    end
end
```

Здесь `b_out` в ветке `else` не назначается. Значит для `b_out` возможен latch.

---

## 4. Вложенные условия с неполным покрытием

Пример:

```verilog
always @(*) begin
    if (en) begin
        if (sel)
            out = a;
    end else begin
        out = b;
    end
end
```

Случай `en = 1` и `sel = 0` оставляет `out` без присваивания.

---

## 5. Сложная combinational логика без default assignments

Когда блок длинный и разветвленный, очень легко забыть одно присваивание. Тогда latch появляется неочевидно.

---

## Очень важный симптом

Если combinational signal должен "сохранять прошлое значение", это почти всегда означает одно из двух:

### Либо

ты на самом деле хотел **регистр**,

### Либо

описание combinational logic неполное.

Именно поэтому фраза вроде:

> "если условие не выполнилось, пусть сигнал останется как был"

в combinational block почти всегда подозрительна.

Потому что "остаться как был" означает память. А память — это уже не чистая combinational logic.

---

## Как правильно избегать unintended latch

Есть несколько базовых приемов.

## 1. Давать default assignment в начале combinational блока

Это один из самых полезных и практичных паттернов.

Пример:

```verilog
always @(*) begin
    out = 1'b0;

    if (sel)
        out = in1;
end
```

Теперь `out` определен всегда:

- по умолчанию `0`;
    
- если `sel=1`, то `in1`.
    

Latch не нужен.

---

## 2. Полностью покрывать `if/else`

Пример:

```verilog
always @(*) begin
    if (sel)
        out = in1;
    else
        out = in0;
end
```

Это полное combinational описание.

---

## 3. Давать `default` в `case`

Пример:

```verilog
always @(*) begin
    case (state)
        2'b00: out = a;
        2'b01: out = b;
        default: out = 1'b0;
    endcase
end
```

Даже если возник "неожиданный" код состояния, `out` остается определенным.

---

## 4. Назначать все выходы во всех ветках

Если блок управляет несколькими сигналами, нужно следить за каждым из них отдельно.

Хороший прием — в начале блока назначить default всем combinational outputs / next-signals.

---

## 5. Использовать `always_comb` вместо старого `always @(*)`, если доступен SystemVerilog

`always_comb` не магически чинит код, но:

- лучше отражает намерение;
    
- инструменты обычно строже проверяют combinational semantics;
    
- код становится дисциплинированнее.

---

## Очень полезный паттерн: default + selective override

Это один из лучших способов писать combinational logic.

Пример:

```verilog
always_comb begin
    next_state = state;
    out_valid   = 1'b0;
    load_cnt    = 1'b0;

    case (state)
        IDLE: begin
            if (start)
                next_state = RUN;
        end

        RUN: begin
            out_valid = 1'b1;
            if (done) begin
                load_cnt   = 1'b1;
                next_state = IDLE;
            end
        end

        default: begin
            next_state = IDLE;
        end
    endcase
end
```

Почему это хорошо:

- у всех сигналов есть default;
    
- дальше идет только переопределение по условиям;
    
- риск latch сильно ниже;
    
- код легче читать.

---

## Latch в FSM coding

FSM — одно из самых частых мест, где unintended latch возникает у начинающих.

Обычно проблема появляется в `next_state` или combinational output logic.

Плохой пример:

```verilog
always @(*) begin
    case (state)
        IDLE: begin
            if (start)
                next_state = RUN;
        end
        RUN: begin
            if (done)
                next_state = IDLE;
        end
    endcase
end
```

Проблема: `next_state` не определен во многих путях.

Правильнее:

```verilog
always @(*) begin
    next_state = state;

    case (state)
        IDLE: begin
            if (start)
                next_state = RUN;
        end
        RUN: begin
            if (done)
                next_state = IDLE;
        end
        default: begin
            next_state = IDLE;
        end
    endcase
end
```

Здесь уже нет необходимости в latch.

---

## Latch и combinational outputs FSM

Тот же принцип касается выходов автомата.

Если output logic описана combinational block’ом, нужно следить, чтобы каждый output был определен всегда.

Иначе автомата вроде бы нет, а память неожиданно появляется через latch.

---

## Почему latch часто считается плохой практикой в FPGA

Формально latch — это допустимый элемент цифровой логики.

Но в FPGA-проектах его обычно избегают, потому что:

- основная методология строится вокруг flip-flops;
    
- timing analysis и поведение проще и предсказуемее без latch;
    
- latch редко нужен намеренно в типичном synchronous RTL;
    
- случайное появление latch почти всегда означает проблему в коде.

То есть вопрос не в том, что latch "запрещен законами природы", а в том, что **в большинстве FPGA RTL-дизайнов он не нужен и чаще вреден**.

---

## Как распознать unintended latch

Есть несколько признаков.

### 1. Предупреждения синтезатора

Например, сообщения о выведенном latch.

Это один из самых прямых сигналов, что combinational block описан неполно.

### 2. Сигнал "помнит" старое значение без явного регистра

На waveform это часто выглядит подозрительно.

### 3. В combinational block есть ветки без назначения

При review такой код сразу должен вызывать вопрос.

### 4. Значение сигнала зависит не только от текущих входов, но и от предыдущего состояния, хотя register не описан

Это красный флаг.

---

## Полезная ментальная проверка

Для любого combinational output спроси себя:

> могу ли я для любого пути исполнения показать, где этому сигналу присваивается значение?

Если нет — почти наверняка есть риск latch.

---

## Частая ошибка мышления

Некоторые пишут combinational block так, как будто это "псевдокод":

```verilog
always @(*) begin
    if (cond1)
        do_something;
    if (cond2)
        do_something_else;
end
```

Но RTL — это не произвольный алгоритмический текст. Это описание железа.

Если ты не описал, что происходит в остальных случаях, инструмент вынужден достраивать поведение аппаратно. И одним из результатов может стать latch.

---

## Хорошая дисциплина combinational coding

Очень полезен такой базовый стиль:

### Для combinational outputs

- default assignments в начале;
    
- потом selective override.
    

### Для `case`

- либо полное покрытие;
    
- либо `default`.
    

### Для `if`

- либо полный `if/else`;
    
- либо заранее заданный default.
    

### Для FSM

- `next_state = state;`
    
- outputs по умолчанию в безопасное состояние.
    

Это почти всегда резко снижает риск unintended latch.

---

## Когда latch может быть намеренным

Теоретически latch можно использовать осознанно.

Но в обычном FPGA RTL flow это редкий и специальный случай.

Если latch действительно нужен, это должно быть:

- явно задумано;
    
- хорошо документировано;
    
- понятно из архитектуры;
    
- не выглядеть как побочный эффект неполного кода.
    

То есть latch не должен появляться "случайно по синтаксису".

---

## `unique case`, `priority case` и latch

Даже если используются специальные формы case/if semantics, они **не отменяют** базового правила:

сигнал должен быть определен во всех путях, если ты хочешь combinational logic без latch.

Эти конструкции помогают формально выразить намерение, но не заменяют полного описания выходов.

---

## Latch и review discipline

Во время review полезно специально искать:

- combinational block без default assignments;
    
- `if` без `else`, если сигнал не инициализирован заранее;
    
- `case` без `default`;
    
- outputs, назначаемые только в части ветвей;
    
- признаки "удержания старого значения" в combinational logic.
    

Это один из простых и очень полезных пунктов RTL review checklist.

---

## Частые ошибки

### Ошибка 1. Забыли назначить сигнал в одной из веток

Самая частая причина latch.

### Ошибка 2. Уверены, что `case` покрывает все варианты, но тип/код состояния шире, чем кажется

Поэтому `default` часто полезен даже при кажущемся полном покрытии.

### Ошибка 3. Назначили часть сигналов, а про остальные забыли

Особенно часто в длинных combinational block’ах.

### Ошибка 4. Хотели "сохранить старое значение" в combinational block

Это почти прямое описание latch.

### Ошибка 5. FSM `next_state` не получает default значения

Тогда latch появляется очень легко.

### Ошибка 6. Путают combinational и sequential роли кода

Если нужна память, обычно нужен clocked register, а не неполный combinational block.

---

## Практические правила

### 1. Для combinational logic всегда думать: значение определено во всех путях?

Это главный вопрос.

### 2. Использовать default assignments в начале блока

Особенно для `next_*`, outputs и control signals.

### 3. Для `case` использовать `default`, если нет очень строгой и осознанной альтернативы

Это помогает и дисциплине, и безопасности.

### 4. Для FSM задавать `next_state = state;` по умолчанию

Один из лучших паттернов.

### 5. Если нужно хранить значение — использовать register в clocked block

А не пытаться получить хранение косвенно через неполную combinational logic.

### 6. Предупреждения о latch воспринимать серьезно

Во многих проектах это не "просто warning", а реальный индикатор дефекта RTL.

---

## Пример плохого кода

```verilog
always @(*) begin
    if (en)
        data_out = data_in;
end
```

Почему плохо:

- при `en=0` нет присваивания;
    
- значит требуется удержание старого значения;
    
- возможен unintended latch.
    

---

## Исправление через `else`

```verilog
always @(*) begin
    if (en)
        data_out = data_in;
    else
        data_out = 8'h00;
end
```

Теперь выход определен всегда.

---

## Исправление через default assignment

```verilog
always @(*) begin
    data_out = 8'h00;

    if (en)
        data_out = data_in;
end
```

Это часто даже удобнее, особенно в сложных блоках.

---

## Пример с несколькими сигналами

Хороший стиль:

```verilog
always_comb begin
    next_state = state;
    out_valid  = 1'b0;
    load_crc   = 1'b0;
    err_flag   = 1'b0;

    case (state)
        IDLE: begin
            if (start)
                next_state = RUN;
        end

        RUN: begin
            out_valid = 1'b1;
            if (crc_error)
                err_flag = 1'b1;
            if (done) begin
                load_crc   = 1'b1;
                next_state = IDLE;
            end
        end

        default: begin
            next_state = IDLE;
        end
    endcase
end
```

Такой стиль хорошо защищает от latch.

---

## Главное, что нужно запомнить

**No unintended latches** — это принцип, по которому combinational logic должна быть описана полностью, чтобы latch не появлялся как побочный эффект неполного RTL.

Latch чаще всего возникает не потому, что инструмент "ошибся", а потому что код говорит:

- в одних случаях выдай новое значение;
    
- в остальных случаях сохрани прежнее.
    

А сохранить прежнее значение — это уже память.

Главный практический вывод:

- если нужна память — описывай register явно;
    
- если нужна combinational logic — назначай значения во всех путях.
    

Золотое правило:

**в combinational block каждый выход должен быть определен всегда.**

---

## Краткая памятка

### Откуда берется unintended latch

- `if` без `else`
    
- `case` без `default`
    
- неполное присваивание сигналов
    
- попытка "удержать старое значение" в combinational block
    

### Как избежать

- default assignments в начале блока
    
- полное покрытие ветвей
    
- `next_state = state;` для FSM
    
- `always_comb` для combinational logic
    

### Когда насторожиться

- synth warning про latch
    
- сигнал помнит прошлое значение без register
    
- не все пути в combinational block назначают выход
    

### Золотое правило

**если это combinational logic, выход не должен зависеть от своей прошлой величины.**

---

## Мини-шаблон для повседневного использования

```verilog
always_comb begin
    next_state = state;
    out_valid  = 1'b0;
    out_data   = '0;

    case (state)
        IDLE: begin
            if (start)
                next_state = RUN;
        end

        RUN: begin
            out_valid = 1'b1;
            out_data  = data_in;
            if (done)
                next_state = IDLE;
        end

        default: begin
            next_state = IDLE;
        end
    endcase
end
```

Этот шаблон отражает один из самых безопасных практических стилей combinational coding без unintended latch.
