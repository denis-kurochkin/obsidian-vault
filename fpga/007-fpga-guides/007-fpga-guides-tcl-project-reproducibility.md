# Tcl project reproducibility в Vivado

## Что это такое

**Tcl project reproducibility** — это способность **воспроизвести один и тот же Vivado-проект и получить эквивалентную сборку**, используя Tcl-скрипты, на:

- другой машине;
- в другой папке;
- после очистки временных файлов;
- другим разработчиком;
- в CI/CD;
- через некоторое время.

Проще говоря, reproducibility означает:

> проект можно не "переносить как папку целиком", а **заново собрать из исходников и Tcl-скрипта**, получив тот же проектовый состав и ожидаемый результат.

В контексте Vivado это одна из главных причин использовать Tcl:

- вместо хранения огромного каталога проекта;
- вместо ручной настройки через GUI;
- вместо зависимости от локальных путей и случайных артефактов.

---

## Интуитивная идея

Есть два подхода к работе с Vivado-проектом.

### Подход 1. "Живой проект в папке"

Ты хранишь почти весь каталог проекта:

- `.xpr`
- `.runs`
- `.cache`
- `.gen`
- `.ip_user_files`
- и много промежуточных файлов.

Минусы:

- проект тяжело переносить;
- в нем много мусора и временных артефактов;
- часть файлов зависит от локальной машины;
- легко получить ситуацию "у меня собирается, у тебя нет".

### Подход 2. "Проект как результат Tcl-сценария"

Ты хранишь:

- RTL;
- XDC;
- Tcl-скрипты;
- возможно `.xci`, `.bd`, конфигурационные файлы;
- список нужных IP и параметров.

А сам проект Vivado **создается заново** командой вроде:

source build.tcl

Это и есть основа reproducibility.

---

## Главная мысль

**Reproducible project** — это когда истинным источником проекта являются:

- исходники,
    
- constraints,
    
- IP-описания,
    
- Tcl-скрипты,
    
- параметры сборки,

а не случайное состояние папки `project/`.

То есть:

**не проектовая папка определяет сборку, а Tcl flow определяет, как проект создать.**

---

## Что именно должно воспроизводиться

У reproducibility есть несколько уровней.

### 1. Воспроизводимость состава проекта

На другой машине должны корректно восстановиться:

- набор HDL-файлов;
    
- top-модуль;
    
- constraints;
    
- IP;
    
- simulation sources;
    
- compile order;
    
- настройки part/board.

### 2. Воспроизводимость flow

Должны одинаково запускаться:

- synthesis;
    
- implementation;
    
- генерация bitstream;
    
- export hardware;
    
- генерация simulation files.

### 3. Воспроизводимость результата

В идеале:

- timing-результаты ожидаемы;
    
- bitstream функционально эквивалентен;
    
- структура проекта не зависит от конкретной машины.

Абсолютно бит-в-бит идентичный результат не всегда гарантируется во всех сценариях, но **логически воспроизводимая сборка** — это уже обязательная цель.

---

## Почему это важно

### 1. Передача проекта другому разработчику

Если коллега не может восстановить проект без твоей папки `.runs`, значит reproducibility слабая.

### 2. Работа через Git

Git хорошо хранит:

- текстовые исходники;
    
- Tcl;
    
- XDC;
    
- конфиги.

Git плохо подходит для хранения:

- кэшей;
    
- временных файлов;
    
- результатов runs;
    
- автоматически сгенерированных больших директорий.

### 3. CI/CD

Автоматическая сборка на сервере возможна только тогда, когда проект можно поднять с нуля скриптом.

### 4. Долгосрочная поддержка

Через полгода ты сам можешь забыть, какие галочки нажимал в GUI.

Tcl фиксирует это явно.

### 5. Снижение числа "магических" проблем

Если вся сборка описана скриптом, меньше скрытых ручных действий.

---

## Что ломает reproducibility чаще всего

## 1. Абсолютные пути

Плохой пример:

```
add_files A:/work/project/src/top.v
```

На другой машине такого пути нет.

Хороший подход:

```
set origin_dir [file normalize [file dirname [info script]]]
add_files [file join $origin_dir src top.v]
```

То есть путь строится **относительно расположения скрипта**, а не относительно диска конкретного пользователя.

---

## 2. Хранение лишнего проектового мусора вместо описания flow

Если в репозитории лежит весь каталог Vivado-проекта, часто непонятно:

- что является исходником истины;
    
- что можно удалять;
    
- какие файлы пересоздаются;
    
- что реально обязательно для сборки.

Воспроизводимость хуже, когда проект зависит от накопившихся артефактов.

---

## 3. Ручные изменения через GUI без фиксации в Tcl

Очень типичная проблема:

- в GUI добавили новый XDC;
    
- поменяли top;
    
- добавили IP;
    
- поменяли strategy;
    
- сохранили проект;
    
- но не обновили Tcl.

После этого скрипт уже не описывает реальное состояние проекта.

---

## 4. Зависимость от локально сгенерированных IP-артефактов

Если сборка требует файлов из:

- `.gen`
    
- `.cache`
    
- `ip_user_files`
    
- `sim`

которые не пересоздаются автоматически, reproducibility страдает.

Настоящий flow должен уметь:

- взять `.xci` или другой источник IP;
    
- регенерировать нужные артефакты;
    
- продолжить сборку без ручного вмешательства.

---

## 5. Неявная зависимость от версии Vivado

Даже хороший Tcl flow может сломаться, если:

- IP обновились;
    
- изменилась версия toolchain;
    
- синтаксис или поведение некоторых команд отличается.

Поэтому reproducibility почти всегда включает фиксацию:

- версии Vivado;
    
- версии IP;
    
- иногда версии board files.

---

## 6. Неявные board files / part settings

Если проект полагается на board definition, которой нет у другого разработчика, проект может не подняться.

Обычно надежнее явно фиксировать:

- `part`;
    
- а если board используется — еще и требования к board files.

---

## 7. Генерация файлов в непредсказуемые места

Если скрипт пишет временные файлы в разные каталоги в зависимости от среды, структуру проекта трудно повторить и отлаживать.

---

## Что обычно хранят в репозитории

Надежный минимальный набор обычно такой:

- RTL-исходники (`.v`, `.sv`, `.vhd`);
    
- constraints (`.xdc`);
    
- Tcl-скрипты (`build.tcl`, `create_project.tcl`, вспомогательные `.tcl`);
    
- IP-описания (`.xci`), если IP создаются через catalog;
    
- block design (`.bd`) или Tcl для его генерации;
    
- файлы параметров/конфигурации;
    
- документацию по версии Vivado.

Иногда также хранят:

- simulation scripts;
    
- mem/init файлы;
    
- patch-файлы;
    
- board-specific configs.

---

## Что обычно НЕ хранят

Обычно не стоит хранить:

- `.runs/`
    
- `.cache/`
    
- `.gen/`
    
- `.Xil/`
    
- `.ip_user_files/`
    
- логи и временные отчеты
    
- автоматически созданные project artifacts, если их можно восстановить

Причина простая: это не исходники истины, а производные артефакты.

---

## Основной принцип reproducible flow

Нужно стремиться к такой модели:

исходники + Tcl + IP descriptions + constraints

↓

create/build script

↓

новый проект Vivado с нуля

↓

synth / impl / bitstream

То есть **папка проекта должна быть воспроизводимым артефактом, а не уникальным объектом**.

---

## Два популярных подхода

## Подход A. Project mode, но проект создается Tcl-скриптом

Скрипт делает примерно следующее:

- `create_project`;
    
- добавляет файлы;
    
- задает top;
    
- подключает XDC;
    
- импортирует/генерирует IP;
    
- создает runs;
    
- запускает synth/impl.

Пример:

```
set origin_dir [file normalize [file dirname [info script]]]
set proj_dir [file join $origin_dir build project]

create_project my_proj $proj_dir -part xcau10p-ffvb676-1-i

add_files [file join $origin_dir src top.v]
add_files [file join $origin_dir src core.v]
add_files -fileset constrs_1 [file join $origin_dir constraints top.xdc]

set_property top top [current_fileset]
update_compile_order -fileset sources_1
```


Плюс этого подхода: он близок к привычному GUI-проекту.

---

## Подход B. Non-project mode

Скрипт читает файлы напрямую и запускает flow без постоянного `.xpr` проекта.

Например:

```
read_verilog [file join $origin_dir src top.v]
read_verilog [file join $origin_dir src core.v]
read_xdc [file join $origin_dir constraints top.xdc]

synth_design -top top -part xcau10p-ffvb676-1-i
opt_design
place_design
route_design
write_bitstream -force [file join $origin_dir build top.bit]
```


Плюсы:

- меньше проектового мусора;
    
- flow часто получается более детерминированным и прозрачным.
    

Минусы:

- не всем удобно;
    
- для некоторых команд/интеграций project mode привычнее.
    

---

## Что значит "write_project_tcl"

Vivado умеет экспортировать проект в Tcl через `write_project_tcl`.

Это полезно как:

- стартовая точка;
    
- способ понять, какие команды нужны для восстановления проекта;
    
- средство миграции от GUI-к-полному Tcl flow.
    

Но важно понимать:

`**write_project_tcl**` **не всегда дает идеальный, чистый, переносимый и поддерживаемый скрипт из коробки.**

Часто после экспорта нужно:

- почистить лишние команды;
    
- убрать абсолютные пути;
    
- отделить обязательное от временного;
    
- проверить, не тянет ли скрипт machine-specific артефакты.
    

То есть это хорошая база, но обычно не финальная архитектура.

---

## Признаки хорошей reproducibility

Проект можно считать хорошо воспроизводимым, если:

### 1. Сборка стартует с чистого каталога

Например, после удаления `build/` проект можно восстановить одной командой.

### 2. Нет зависимости от локальных абсолютных путей

Все пути относительные или вычисляются от `origin_dir`.

### 3. Скрипт сам добавляет все нужные файлы

Ничего не требуется добавлять руками в GUI.

### 4. IP можно регенерировать

Скрипт не требует уже сгенерированных вручную промежуточных папок.

### 5. Версия Vivado зафиксирована или явно указана

Минимум в README, лучше в проверке скрипта.

### 6. После клона репозитория новый разработчик понимает, как собрать проект

Идеально, если инструкция помещается в несколько строк.

## Практический шаблон структуры репозитория

project_root/

src/

rtl/

constraints/

ip/

bd/

tcl/

create_project.tcl

build.tcl

ip.tcl

debug_constraints.tcl

sim/

scripts/

build/

README.md

.gitignore

Идея:

- исходники и описания — в репозитории;
    
- сборочные артефакты — в `build/`;
    
- проект можно снести и пересоздать.
    

---

## Полезные приемы

## 1. Использовать `origin_dir`

Это один из базовых элементов переносимости.

set origin_dir [file normalize [file dirname [info script]]]

Если `build.tcl` лежит, например, в `tcl/`, можно подняться на уровень выше:

set origin_dir [file normalize [file join [file dirname [info script]] ..]]

---

## 2. Хранить build artifacts отдельно

Например:

set build_dir [file join $origin_dir build vivado]

file mkdir $build_dir

Тогда всегда ясно, что можно удалить без потери исходников.

---

## 3. Явно задавать part

Например:

create_project my_proj $build_dir -part xc7a100tfgg484-2

Это надежнее, чем надеяться на GUI-состояние.

---

## 4. Явно задавать top

set_property top top_module [current_fileset]

Иначе возможны неожиданности при изменении состава файлов.

---

## 5. Обновлять compile order в скрипте

update_compile_order -fileset sources_1

update_compile_order -fileset sim_1

---

## 6. Проверять наличие обязательных файлов

proc require_file {path} {

if {![file exists $path]} {

error "Required file not found: $path"

}

}

Это лучше, чем падать позже в неочевидном месте.

---

## 7. Разделять создание проекта и запуск сборки

Например:

- `create_project.tcl` — только создает и настраивает проект;
    
- `build.tcl` — вызывает создание и запускает runs.
    

Так проще отлаживать.

---

## 8. Логировать ключевые параметры

puts "INFO: Origin dir = $origin_dir"

puts "INFO: Build dir = $build_dir"

puts "INFO: Part = $part_name"

puts "INFO: Top = $top_name"

Это сильно помогает в batch mode.

---

## 9. Проверять версию Vivado

Пример простой проверки:

set expected_version "2024.2"

set current_version [version -short]

if {$current_version ne $expected_version} {

puts "WARNING: Expected Vivado $expected_version, got $current_version"

}

Для критичных flow можно даже сделать `error`.

---

## 10. Делать flow идемпотентным

Желательно, чтобы повторный запуск:

- не ломал проект;
    
- либо корректно пересоздавал build dir;
    
- либо явно очищал старое состояние.
    

Например:

if {[file exists $build_dir]} {

file delete -force $build_dir

}

file mkdir $build_dir

Но это нужно делать осознанно, чтобы не удалить важное.

---

## Пример простого reproducible create_project.tcl

set origin_dir [file normalize [file join [file dirname [info script]] ..]]

set build_dir [file join $origin_dir build project]

set part_name xc7a200tfbg484-2

set top_name top

  

if {[file exists $build_dir]} {

file delete -force $build_dir

}

file mkdir $build_dir

  

create_project my_proj $build_dir -part $part_name

  

proc require_file {path} {

if {![file exists $path]} {

error "Required file not found: $path"

}

}

  

set rtl_files [list \

[file join $origin_dir src top.v] \

[file join $origin_dir src core.v] \

[file join $origin_dir src uart.v] \

]

  

foreach f $rtl_files {

require_file $f

add_files $f

}

  

set xdc_file [file join $origin_dir constraints top.xdc]

require_file $xdc_file

add_files -fileset constrs_1 $xdc_file

  

set_property top $top_name [current_fileset]

update_compile_order -fileset sources_1

update_compile_order -fileset sim_1

  

puts "INFO: Project created successfully"

Это уже заметно лучше, чем ручная настройка через GUI.

---

## Пример build.tcl

set origin_dir [file normalize [file join [file dirname [info script]] ..]]

source [file join $origin_dir tcl create_project.tcl]

  

launch_runs synth_1 -jobs 8

wait_on_run synth_1

  

launch_runs impl_1 -to_step write_bitstream -jobs 8

wait_on_run impl_1

  

puts "INFO: Build finished"

Такой flow уже можно запускать одинаково у всех участников команды.

---

## Что делать с IP

IP — одна из самых частых причин плохой воспроизводимости.

Обычно хороший подход такой:

- хранить `.xci`;
    
- при необходимости вызывать генерацию IP через Tcl;
    
- не полагаться на старые `.gen` и `.cache` как на обязательный источник истины.
    

Типичный принцип:

.xci хранится

↓

при сборке IP при необходимости регенерируется

↓

дальше идет synth/impl

Если IP создается полностью из Tcl, это даже лучше: тогда проект вообще не зависит от заранее сохраненного project state.

---

## Что делать с Block Design

Для BD есть два распространенных пути:

### Вариант 1

Хранить `.bd` и восстанавливать проект с ним.

### Вариант 2

Экспортировать Tcl, который заново создает block design.

Второй вариант часто лучше для reproducibility, но он требует аккуратного сопровождения.

---

## GUI и reproducibility

GUI сам по себе не зло. Проблема не в GUI, а в том, что ручные действия часто не фиксируются явно.

Нормальная практика может быть такой:

- разработка и исследование — через GUI;
    
- итоговая конфигурация фиксируется в Tcl;
    
- воспроизводимая сборка выполняется скриптом.
    

То есть GUI можно использовать как рабочий инструмент, но **источник истины должен быть в текстовых файлах и скриптах**.

---

## Типовые ошибки мышления

### Ошибка 1. "У меня же есть .xpr, значит проект воспроизводим"

Не обязательно.

Если `.xpr` зависит от локальных путей, внешних board files или незафиксированных артефактов, воспроизводимость слабая.

### Ошибка 2. "Сохраню весь проект целиком в Git — и проблема решена"

Обычно нет.

Это лишь переносит мусор и машинно-зависимое состояние в репозиторий.

### Ошибка 3. "write_project_tcl все решает автоматически"

Нет. Это хорошая заготовка, но ее почти всегда нужно проверять и чистить.

### Ошибка 4. "Если проект собрался один раз, значит он reproducible"

Один успешный запуск не доказывает воспроизводимость.

Настоящая проверка — это:

- новый каталог;
    
- желательно другая машина;
    
- запуск только по инструкции и Tcl.
    

---

## Практический тест на reproducibility

Хороший быстрый тест такой:

1. Удалить каталог сборки.
    
2. Клонировать проект в новый путь.
    
3. Запустить только документированный Tcl flow.
    
4. Проверить, что:
    
    - проект создается;
        
    - IP доступны;
        
    - synth/impl идут без ручных действий;
        
    - bitstream получается.
        

Если на любом шаге требуется "еще вот это руками нажать" — reproducibility неполная.

---

## Рекомендуемая философия

Лучше всего смотреть на Vivado-проект так:

> `project/` — это временный build artifact,  
> а `src/ + constraints/ + ip/ + tcl/` — это настоящий проект.

Это очень полезная мысль. Она меняет архитектуру репозитория в лучшую сторону.

---

## Когда reproducibility особенно критична

Особенно важно делать воспроизводимый Tcl flow, если:

- в проекте несколько разработчиков;
    
- проект идет в производство;
    
- есть CI/CD;
    
- используются IP и BD;
    
- проект живет долго;
    
- требуется передача другой команде или заказчику.
    

---

## Когда можно начать с упрощенного варианта

Если проект маленький, можно начать с минимального набора:

- один `create_project.tcl`;
    
- все пути относительные;
    
- явный `part`, `top`, `add_files`, `add_files -fileset constrs_1`;
    
- `.gitignore` для build artifacts.
    

Даже это уже дает большой выигрыш по сравнению с полностью ручным flow.

## Главное, что нужно запомнить

**Tcl project reproducibility** в Vivado — это подход, при котором проект должен надежно восстанавливаться из исходников и Tcl-скриптов, а не из случайного состояния рабочей директории.

Главная цель:

- любой участник команды может пересоздать проект и сборку с нуля.
    

Главные враги:

- абсолютные пути;
    
- ручные GUI-изменения без фиксации;
    
- зависимость от кэшей и `.runs`;
    
- незафиксированная версия Vivado;
    
- неявные IP/board-зависимости.
    

Главный принцип:

**если build directory можно удалить и потом полностью восстановить Tcl-скриптом, значит reproducibility на хорошем уровне.**

---

## Краткая памятка

### Что это

Воспроизводимость Vivado-проекта через Tcl-сценарии.

### Что должно быть источником истины

- RTL
    
- XDC
    
- IP descriptions
    
- Tcl scripts
    
- config files
    

### Что не должно быть источником истины

- `.runs`
    
- `.cache`
    
- `.gen`
    
- `.Xil`
    
- случайное GUI-состояние
    

### Золотые правила

- использовать относительные пути;
    
- фиксировать `part` и `top`;
    
- не полагаться на build artifacts;
    
- регенерировать IP;
    
- документировать версию Vivado;
    
- проверять сборку с нуля.

## Мини-шаблон для повседневного использования

set origin_dir [file normalize [file join [file dirname [info script]] ..]]

set build_dir [file join $origin_dir build project]

set part_name xc7a100tfgg484-2

set top_name top

  

proc require_file {path} {

if {![file exists $path]} {

error "Required file not found: $path"

}

}

  

if {[file exists $build_dir]} {

file delete -force $build_dir

}

file mkdir $build_dir

  

create_project my_proj $build_dir -part $part_name

  

set rtl_files [list \

[file join $origin_dir src top.v] \

[file join $origin_dir src core.v] \

]

  

foreach f $rtl_files {

require_file $f

add_files $f

}

  

set xdc_file [file join $origin_dir constraints top.xdc]

require_file $xdc_file

add_files -fileset constrs_1 $xdc_file

  

set_property top $top_name [current_fileset]

update_compile_order -fileset sources_1

  

puts "INFO: Reproducible project created"

Это не идеальная универсальная система, но это уже правильная база для воспроизводимого Tcl flow.
