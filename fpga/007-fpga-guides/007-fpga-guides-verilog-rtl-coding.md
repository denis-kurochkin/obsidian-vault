### Как написать синтезируемый код на Verilog?
- Убрать ненужные защелки
	Наличие комбинационной логики с обратной связью.
	- **В операторе присваивания:** Когда по ошибке вы повторно используете выходную переменную в качестве одного из операндов.
	- **В блоке always** отсутствует логика, определяющая, что делать при высоком значении reset. По умолчанию c сохраняет свое значение, образуя защелку (ниже приведен RTL-код, созданный с помощью программного обеспечения Vivado Simulation от Xilinx для первого экземпляра кода).
	
- Ошибка ввода с несколькими управляющими воздействиями
	**В блоке always:** управление регистром в нескольких блоках always создает эту проблему. Симуляторы отдают приоритет одному из них.
	
- Критическая ошибка выполнения
	- Выход за пределы допустимого диапазона доступа к памяти:** доступ к несуществующей переменной вектора reg.  
	    Пример: (reg [2:0] mem [12:0]; assign output = mem[13];). Переменная mem[13] здесь не существует, следовательно, это выход за пределы допустимого диапазона доступа к памяти.
    - **Застрял в замкнутом круге:** из-за циклической зависимости.
    - **Блокировка "Always" без элемента управления событиями:** Если блокировка "Always" не содержит списка конфиденциальных данных или элемента управления событиями, она будет выполняться бесконечно.
	
- Использование начальных блоков, системных задач и функций ($finish, $display и т. д.)

## Предотвращение непреднамеренных защелок

- Избегание **защелок** в коде Verilog — один из важнейших шагов в повышении **производительности, надежности и точности синхронизации FPGA** . Защелки (особенно _непреднамеренные_ ) вызывают непредсказуемые временные искажения, усложняют синтез и потребляют больше ресурсов. Ниже приводится подробный анализ **причин** их возникновения, **способов обнаружения** и **предотвращения** — с примерами.

- Почему защелки представляют проблему
	- **Непредсказуемость временных характеристик** – Защелки чувствительны к уровню сигнала (а не срабатывают по фронту), поэтому анализ временных характеристик становится сложным и затрудняет оптимизацию с помощью инструментов синтеза.
	- **Сбои и состояния гонки** — Защелки могут распространять нестабильные сигналы
	- **Неэффективное использование ресурсов** – ПЛИС оптимизированы для триггеров, а не для прозрачных защелок.
	- **Сложнее соблюдать временные требования** – конструкции на основе защелок затрудняют статический анализ временных характеристик (STA), что приводит к снижению максимальной частоты (Fmax).
	
- Почему защелки (непреднамеренно) распознаются косвенно
	- Защелка считается установленной, если сигнал в **комбинационном блоке «всегда»** не **назначается при всех возможных условиях** .
	- Пример непреднамеренного срабатывания защелки:
```
	module m(   
	    input d,enable,   
	    output reg q   
	    );   
	    always @(*) begin   
	        if (enable)   
	            q = d;   
	        // Отсутствует "else" - блокировка определяется   
	    end   
	endmodule
	
Здесь, когда  enable значение равно 0, q оно сохраняет свое предыдущее значение, что подразумевает **хранение** → **защелка**.
```
	 
- Как избежать защелок (рекомендации)
	- Присвойте значения по умолчанию в комбинационных блоках.
		- В начале блока всегда инициализируйте выходные значения значением по умолчанию.
```
			module m(  
			вход d,enable,  
			output reg q  
			);  
			  
			always @(*) 
			begin  
				q = 0 ;  
				if (enable)  
					q = d;  
			end  
			endmodule
			
			Теперь `q`назначается во всех случаях → **без защелки** .
```
