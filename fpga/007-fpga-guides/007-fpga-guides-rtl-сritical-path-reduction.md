## Critical path reduction в FPGA Digital Design/RTL

**Critical path reduction** — это уменьшение задержки по самому длинному временному пути в дизайне.  
Именно **critical path** чаще всего определяет, закроется timing или нет, и какую максимальную частоту сможет выдержать FPGA.

Если сказать очень практично:

- есть регистр-источник,
- дальше сигнал проходит через LUT-логику, mux, carry-chain, routing,
- и приходит на вход следующего регистра.

Если этот путь длиннее, чем период такта, timing не закроется.  
Значит, вся задача critical path reduction сводится к одному вопросу:

**как сделать самый длинный путь короче или легче для реализации?**

---

# 1. Что такое critical path на практике

В учебном смысле critical path — это путь с наибольшей задержкой между двумя синхронными точками.

В реальном FPGA-проекте в эту задержку обычно входят:

- **clock-to-out** исходного регистра,
- **комбинаторная логика**,
- **routing delay**,
- **setup time** целевого регистра.

И часто неожиданно оказывается, что проблема не только в “слишком сложной логике”, а в комбинации:

- много уровней LUT,
- длинная цепочка mux,
- широкий comparator,
- высокий fanout,
- неудобное placement,
- длинный routing.

Поэтому critical path reduction — это не только “переписать always_comb красивее”, а работа на уровне **архитектуры, RTL и физической реализации**.

---

# 2. Почему critical path появляется

У критического пути почти всегда есть одна или несколько из этих причин.

## 2.1 Слишком много логики за один такт

Например, в одном цикле ты пытаешься:

- декодировать команду,
- проверить несколько условий,
- вычислить адрес,
- сделать арифметику,
- выбрать один из результатов,
- сформировать выход.

RTL может выглядеть компактно, но для FPGA это может оказаться слишком тяжелым участком между регистрами.

---

## 2.2 Длинные цепочки выбора

Типичные источники:

- `if / else if / else if ...`
- большие `case`
- каскадные тернарные операторы
- приоритетные декодеры
- длинные mux trees

Особенно опасно, когда выбор идет **последовательно**, а не как сбалансированное дерево.

---

## 2.3 Сложная арифметика

Например:

- несколько сумматоров подряд,
- adder + comparator + saturate + mux в одном такте,
- умножение плюс дополнительная постобработка,
- длинные carry-chain path.

Даже если FPGA хорошо умеет арифметику, слишком много зависимых операций в одном cycle быстро создает критический путь.

---

## 2.4 Wide control logic

Очень часто критический путь идет не по данным, а по управлению:

- next-state logic FSM,
- `ready`/`valid` backpressure,
- enable conditions,
- глобальные qualifiers,
- logic “можно ли сейчас сделать операцию”.

Это особенно неприятно, потому что такие цепочки часто проходят через много модулей.

---

## 2.5 Высокий fanout

Даже простой сигнал может стать частью критического пути, если он управляет большим количеством логики или регистров:

- reset,
- clock enable,
- valid,
- mode,
- decode result,
- one-hot control.

Тогда проблема уже не в глубине LUT, а в том, что сеть тяжело развести.

---

## 2.6 Неудачная физическая реализация

Иногда RTL вроде разумный, но после synthesis/place&route path становится длинным из-за:

- плохого placement,
- сильной congestion,
- логики, разбросанной по кристаллу,
- пересечения иерархий,
- неудачного replication/packing.

Поэтому critical path reduction нельзя сводить только к тексту HDL.

---

# 3. Главная идея reduction: уменьшить работу между регистрами

Все методы обычно сводятся к одному из пяти приемов:

1. **разделить путь регистрами**
2. **упростить саму логику**
3. **сбалансировать вычисление**
4. **уменьшить fanout и routing pressure**
5. **лучше согласовать RTL с архитектурой FPGA**

Ниже — подробнее.

---

# 4. Самый сильный метод: pipeline

Это почти всегда первый и самый мощный способ.

## Идея

Если путь слишком длинный, надо не заставлять FPGA сделать его за один такт, а разбить на несколько стадий.

Вместо такого:

- decode
- arithmetic
- compare
- select
- output

за один цикл,

делают:

- **stage 1**: decode / подготовка
- **stage 2**: arithmetic
- **stage 3**: compare / select
- **stage 4**: output register

## Что это дает

- уменьшается combinational depth между регистрами,
- упрощается routing,
- повышается Fmax,
- легче закрыть timing.

## Что это стоит

- растет latency,
- нужно протягивать control signals по pipeline,
- интерфейс может усложниться.

Но для FPGA это обычно самый правильный компромисс.

### Практическое правило

Если критический путь длинный и архитектурно действительно делает “слишком много за такт”, не пытайся лечить его мелким переписыванием RTL.  
Скорее всего, нужен **pipeline**, а не косметика.

---

# 5. Logic restructuring: переписать логику так, чтобы она была легче

Иногда pipeline добавить нельзя или не хочется. Тогда следующий уровень — перестроение самой логики.

## 5.1 Убирать последовательные цепочки условий

Плохо:

if (a) y = x0;  
else if (b) y = x1;  
else if (c) y = x2;  
else if (d) y = x3;  
else y = x4;

Такой код может породить длинную приоритетную цепочку.

Иногда лучше:

- вынести условия заранее,
- сделать one-hot decode,
- разбить выбор на уровни,
- использовать более параллельную структуру.

## 5.2 Балансировать дерево выбора

Если есть выбор из многих источников, выгоднее не строить длинную линейную цепь, а сделать дерево 2:1 или staged selection.

## 5.3 Выносить общие подвыражения

Если одно и то же сложное условие повторяется в нескольких местах, полезно:

- вычислить его один раз,
- зарегистрировать при необходимости,
- использовать уже готовый локальный control signal.

Это может сократить и глубину, и routing pressure.

## 5.4 Уменьшать зависимость операций

Плохо, когда результат одной операции сразу идет в следующую, а та — в следующую:

- add
- compare
- saturate
- mux

Это последовательная зависимость.  
Иногда часть можно вычислить заранее или параллельно.

---

# 6. Arithmetic path reduction

Арифметика — один из главных источников критических путей.

## 6.1 Не делать слишком много зависимых операций в одном цикле

Например, вместо:

- `sum = a + b + c + d`
- потом сравнение
- потом clamp

лучше разбить на стадии:

- stage 1: `ab = a + b`, `cd = c + d`
- stage 2: `sum = ab + cd`
- stage 3: compare/clamp

## 6.2 Балансировать reduction tree

Если нужно суммировать много операндов, линейная цепочка хуже, чем дерево.

Плохо:

- `((((a+b)+c)+d)+e)+f`

Лучше:

- `(a+b)`, `(c+d)`, `(e+f)`
- потом суммировать результаты

## 6.3 Использовать DSP осознанно

Если операция хорошо ложится в DSP block, обычно это лучше, чем строить широкую арифметику на LUT.

Но важно понимать:

- DSP тоже имеет pipeline options,
- без внутренних регистров даже DSP-путь может стать тяжелым,
- лучший результат часто получается именно с включенной DSP pipeline.

## 6.4 Осторожно с wide compare

Широкие сравнения, особенно если они идут вместе с арифметикой и selection, часто становятся тяжелыми.  
Иногда полезно:

- сравнивать по частям,
- вынести compare в отдельный stage,
- сначала вычислить признаки, потом решение.

---

# 7. Control path reduction

Очень частая ошибка — сосредоточиться только на datapath, хотя critical path идет по control.

## 7.1 FSM next-state logic

Сложные условия перехода могут быть тяжелее самих данных.  
Полезные приемы:

- упрощать state machine,
- делить большую FSM на несколько локальных,
- использовать predecode,
- регистрировать промежуточные флаги условий,
- не пытаться в одной FSM держать всю глобальную политику системы.

## 7.2 Ready/valid chains

Очень опасный класс путей — когда `ready` комбинаторно протягивается через несколько модулей назад.

Например:

- downstream `ready`
- влияет на middle block
- тот меняет upstream `ready`
- и так через длинную цепочку

Это легко создает критический путь.

Лекарства:

- register slice,
- skid buffer,
- stage boundary,
- локальный buffering.

Во многих потоковых архитектурах именно это дает самый заметный выигрыш.

## 7.3 Глобальные enable signals

Если один control signal решает слишком много всего сразу, его fanout и routing могут стать критическими.  
Тогда помогает:

- локализация enable,
- replication,
- staged control,
- разбиение одного глобального признака на несколько локальных.

---

# 8. Fanout reduction

Иногда path логически не очень глубокий, но timing плохой из-за того, что сигнал раздается слишком широко.

Примеры:

- reset идет на полдизайна,
- mode signal управляет множеством mux,
- valid/enable используется во многих местах,
- один decode result расклеен по большой области.

### Что делать

- регистрировать сигнал ближе к месту использования,
- делать локальные копии,
- дробить управление на иерархические уровни,
- избегать лишне глобальной логики,
- иногда дать инструменту возможность replication.

Особенно важно понимать, что **fanout problem часто выглядит как logic problem**, хотя корень — именно в сети распространения сигнала.

---

# 9. Routing-oriented reduction

В FPGA большая часть задержки может сидеть не в LUT, а в routing.

Поэтому critical path reduction часто достигается не изменением “математики”, а улучшением физической локальности.

## Помогает:

- держать связанные stage рядом архитектурно,
- не размазывать одну функцию по многим модулям без нужды,
- уменьшать cross-hierarchy critical control,
- регистрировать границы между крупными блоками,
- не строить огромные комбинаторные связи через весь дизайн.

Хорошее RTL обычно локализует вычисление:

- данные обрабатываются рядом,
- control рядом с тем stage, который им пользуется,
- глобальных мгновенных зависимостей мало.

---

# 10. Register balancing

Это метод, при котором вычисление распределяется между стадиями более равномерно.

Плохая картина:

- stage A почти пустой,
- stage B огромный и критичный,
- stage C снова легкий.

Хорошая картина:

- работа распределена примерно равномерно,
- ни один stage не перегружен.

## На практике это значит:

- перенести часть вычисления раньше,
- перенести часть позже,
- зарегистрировать промежуточный результат,
- выровнять control pipeline с data pipeline.

Иногда один лишний регистр в правильном месте дает гораздо больше, чем сложный рефакторинг условий.

---

# 11. Retiming и почему на него нельзя полностью рассчитывать

Синтез/implementation tools иногда умеют **retiming** — перенос регистров через часть логики автоматически.

Но в реальных FPGA-дизайнах полностью полагаться на это нельзя, потому что:

- не вся логика подходит для ретайминга,
- reset/enable мешают,
- иерархия и специальные блоки мешают,
- tool может не понять архитектурный смысл,
- лучший pipeline boundary обычно знает разработчик, а не инструмент.

Поэтому retiming — приятный бонус, но не замена осознанному pipeline/stage design.

---

# 12. Как читать проблему правильно: от report к исправлению

Когда timing report показывает критический путь, важно не просто видеть “path long”, а понять, **почему** он длинный.

Нужно спрашивать:

- path data-heavy или control-heavy?
- проблема в LUT levels или routing?
- это один локальный блок или путь проходит через несколько модулей?
- есть ли явная natural stage boundary?
- этот путь можно pipeline’ить?
- fanout является главным фактором?
- это повторяющийся тип проблемы или единичный случай?

То есть correction должен исходить из природы path, а не из желания “что-нибудь переписать”.

---

# 13. Что часто не помогает

Есть несколько типичных ложных решений.

## 13.1 “Сделаю код короче”

Короткий RTL не обязательно означает короткий path.

## 13.2 “Разобью always_comb на несколько always_comb”

Если между ними нет регистра, для timing это все еще одна комбинаторная цепочка.

## 13.3 “Инструмент сам оптимизирует”

Иногда — да, но если архитектурно за цикл делается слишком много, tool не сотворит чудо.

## 13.4 “Снижу constraints”

Это не решение. Дизайн не стал быстрее, просто требования стали мягче.

## 13.5 “Добавлю false path”

Если path реальный и рабочий, маскировать его constraint’ом опасно.

---

# 14. Очень типичные источники критических путей в RTL

Ниже — список, который часто встречается в FPGA-дизайнах.

## 14.1 Long if-else priority logic

Особенно в control path.

## 14.2 Широкие mux

Например, выбор одного из многих источников данных в одном такте.

## 14.3 Ready chain across modules

Один из классических случаев в streaming designs.

## 14.4 Large compare + arithmetic + select

Например, адресная логика, saturating arithmetic, scoreboarding.

## 14.5 Большая next-state logic

Слишком “умная” FSM.

## 14.6 Shared global qualifiers

Один глобальный признак участвует в очень многих решениях.

## 14.7 CRC/check/parity + decision in one cycle

Даже если функция логически простая, combined path может стать тяжелым.

---

# 15. Практическая стратегия reduction

Хорошая рабочая последовательность обычно такая.

## Шаг 1. Найти настоящий path

Не лечить абстрактный “дизайн медленный”, а посмотреть timing report и понять источник.

## Шаг 2. Классифицировать path

Это:

- datapath,
- control path,
- fanout path,
- routing-heavy path,
- arithmetic path,
- handshake path?

## Шаг 3. Выбрать правильный класс решения

- pipeline,
- logic restructure,
- register slice,
- fanout reduction,
- arithmetic balancing,
- local buffering,
- stage rebalance.

## Шаг 4. Исправлять минимально, но архитектурно

Не переписывать весь блок без нужды.  
Найти естественную границу, где путь можно разорвать.

## Шаг 5. Перепроверить, не создал ли новый критический путь

После фикса старый path может исчезнуть, но новый станет главным.

# 16. Пример мысленного разбора

Допустим, есть path:

- регистр входа команды
- decode opcode
- проверка состояния FSM
- расчет адреса
- compare against limit
- выбор источника данных
- запись в output register

Это почти textbook-пример перегруженного цикла.

### Возможные исправления:

- stage 1: decode + state qualification
- stage 2: address calc
- stage 3: limit compare + source select
- stage 4: output register

Если latency критична, можно искать компромисс:

- decode вынести заранее,
- compare сделать на предыдущем цикле,
- source select упростить,
- часть control зарегистрировать отдельно.

Главная мысль:  
**не пытаться сделать весь смысл операции “в одном вдохе”, если timing уже показывает, что железо не успевает**.

---

# 17. Trade-off: latency vs Fmax vs complexity

Critical path reduction почти всегда означает компромисс.

## Больше pipeline

Плюсы:

- выше Fmax,
- легче timing,
- лучше масштабирование.

Минусы:

- больше latency,
- сложнее control alignment,
- больше регистров.

## Меньше pipeline

Плюсы:

- ниже latency,
- проще на уровне “идеальной” функциональной модели.

Минусы:

- тяжелее timing,
- хуже переносимость на более высокие частоты,
- больше риск неприятных критических путей.

В FPGA на практике очень часто выигрывает подход:  
**лучше чуть больше latency, но clean stage boundaries и хороший timing**.

---

# 18. Полезный инженерный mindset

Хороший разработчик RTL постоянно задает себе вопрос:

**“Эта логика действительно должна завершиться за один такт?”**

И еще три подвопроса:

- можно ли посчитать часть раньше?
- можно ли посчитать часть позже?
- можно ли зарегистрировать промежуточный результат?

Именно это и есть основа critical path reduction mindset.

---

## Краткое резюме

**Critical path reduction** — это уменьшение задержки по самому длинному пути, который ограничивает частоту дизайна.

Основные способы:

- **pipeline**
- **register balancing**
- **logic restructuring**
- **arithmetic tree balancing**
- **fanout reduction**
- **register slice / buffering**
- **локализация control и data**
- **уменьшение длинных routing-зависимостей**

Главный практический вывод:

**если путь длинный архитектурно, его надо лечить архитектурно, а не косметически.**  
То есть не просто “переписать код красивее”, а изменить структуру выполнения логики между регистрами.
