## Address space layout

**Address space layout** — это то, **как именно разбито системное адресное пространство на области**: память, периферия, регистры IP, окна мостов, debug-диапазоны, reserved/hole области и т.д. Если **address mapping** отвечает на вопрос _«какой адрес ведет к какому slave»_, то **address space layout** отвечает на вопрос _«как всю карту адресов организовать так, чтобы она была понятной, масштабируемой и безопасной для интеграции»_. В системах FPGA/SoC это напрямую связано с тем, что у master есть свое address space, а slave-сегменты становятся видимыми в этом пространстве по назначенным base address и range.

### 1. Что именно проектируется

Когда говорят про layout, речь обычно не о внутренних регистрах одного IP, а о **структуре всей карты адресов платформы**. То есть заранее решают, где будут жить крупные классы ресурсов, например:

- low-speed периферия
- control/status регистры accelerator’ов
- on-chip memory
- DDR windows
- bridge/aperture области
- debug/service blocks
- reserved gaps под будущее расширение

Идея та же, что и в инструментах Vivado/Platform Designer: у master’а есть глобально видимая карта, а отдельные slave получают в ней свои диапазоны.

### 2. Зачем layout нужен отдельно от самого mapping

Формально можно просто раздать свободные base address по мере добавления блоков. Но это обычно быстро приводит к хаотичной карте адресов, которую трудно расширять и трудно сопровождать в software. AMD прямо подчеркивает, что по мере роста сложности дизайна адресную карту нужно **лучше понимать и визуализировать как системную карту**, а не как набор разрозненных назначений. Intel, со своей стороны, делает Address Map отдельной точкой настройки master↔slave связей и диапазонов адресов.

Практически layout нужен для трех вещей. Во-первых, чтобы **decode interconnect’а был однозначным и эффективным**. Во-вторых, чтобы разным master’ам можно было дать разную видимость ресурсов. В-третьих, чтобы software получал предсказуемую и стабильную memory map. AMD в Address Editor различает `Assigned`, `Unassigned` и `Excluded`, то есть layout влияет не только на адрес, но и на саму доступность slave для master’а.

### 3. Базовая хорошая структура layout

На практике хороший address space layout обычно строят **крупными зонами**, а не по одному блоку в случайные места. Например, отдельно выделяют:

- область периферии
- область control/status регистров PL/IP
- область большой памяти
- область локальной on-chip memory
- bridge windows в другие сегменты
- reserved области под рост

Это не жесткое требование стандарта, а архитектурная дисциплина. Она помогает и hardware, и software, потому что по верхним битам адреса уже видно класс ресурса, а interconnect получает естественные крупные диапазоны для decode. Такая логика хорошо согласуется с тем, как Vivado и Platform Designer показывают address map: как разбиение адресного пространства сети/мастера на slave-области.

### 4. Base, range, alignment

Любой layout в итоге сводится к выбору для каждого сегмента:

- **base address**
- **range / span**
- иногда еще **visibility** по master’ам

Очень важный практический момент: Intel прямо требует, чтобы base address slave был **кратен address span** этого компонента, потому что это дает более эффективную логику address decode и помогает достичь лучшего `fMAX`. Это хороший системный ориентир и вне конкретного инструмента: layout, в котором диапазоны выровнены крупно и логично, почти всегда легче декодировать и сопровождать.

Отсюда следует полезное инженерное правило: лучше мыслить не “у этого IP 44 байта регистров, значит дам ему ближайшие 44 байта”, а “этому классу блоков выделяю понятный выровненный регион, а внутри уже использую offsets”. Это уже архитектурный layout, а не локальная экономия адресов. Intel и AMD оба фактически работают именно с такими сущностями: base + span/range + принадлежность segment/address space.

### 5. Layout и разные master’ы

Очень важная системная мысль: **layout не обязан быть одинаково виден всем master’ам**. Один и тот же slave может быть виден одному master’у по одному диапазону, другому — по другому, а третьему — вообще быть скрыт. Intel прямо пишет, что два master’а могут обращаться к одному и тому же slave **по разным адресам**; AMD отдельно поддерживает исключение slave из видимости конкретного master’а через `Excluded`.

Из этого следует, что address space layout — это не просто “глобальный список адресов”, а **проектирование нескольких представлений системы**:

- CPU view
- DMA view
- debug view
- accelerator view

Именно поэтому на уровне SoC layout лучше обсуждать вместе с interconnect и правами доступа, а не только вместе с регистровой картой одного IP.

### 6. Хороший layout обычно иерархичен

Когда система растет, полезно думать о layout не как о плоском наборе адресов, а как о **иерархии областей**:

- большая верхнеуровневая зона
- внутри нее подзоны по классам устройств
- внутри подзоны — отдельные slave
- внутри slave — offsets регистров или memory pages

Это особенно естественно, если в системе есть address network, bridge или subsystem. AMD показывает address map именно как layout AXI slave’ов в **network address space**, а не только как линейный список регистров.

Иерархический layout удобен потому, что позволяет менять внутренности подсистемы, не ломая внешний контракт. Снаружи сохраняется большая region/base, а внутри можно перераспределять offsets между внутренними IP.

### 7. Reserve gaps и “дырки” — это не всегда плохо

В system-level layout **свободные окна** часто полезны. Они нужны для:

- будущего расширения подсистемы
- смены размера memory window
- версии 2 того же IP
- выравнивания decode
- упрощения software map

То есть пустая область между двумя диапазонами — не обязательно ошибка. Ошибка начинается тогда, когда software считает ее валидной памятью, а hardware на самом деле туда не маршрутизирует доступ. Тогда это уже undefined region / decode error scenario. В Platform Designer и подобных системах такие области естественно возникают между назначенными spans.

### 8. Какие layout-паттерны обычно удачны

Есть несколько устойчиво удачных решений.

**Функциональное зонирование.**  
Отдельно память, отдельно периферия, отдельно service/debug. Это улучшает читаемость карты и сокращает хаос при росте системы. Такая организация хорошо согласуется с тем, что инструменты показывают карту как набор системных slave-region’ов.

**Крупное выравнивание.**  
Лучше давать выровненные, легко читаемые диапазоны, чем уплотнять карту адресов до минимума. Для Intel это еще и прямо связано с эффективностью decode и `fMAX`.

**Место под рост.**  
Если рядом с accelerator control block оставить запас адресов, следующая версия IP не потребует передвигать всю карту и переписывать software.

**Стабильный software contract.**  
Если base addresses часто меняются между ревизиями hardware, то драйверы, linker regions, device tree и диагностика начинают “плыть”. Платформенные flow AMD и Intel явно завязаны на перенос аппаратной конфигурации в software artifacts: XSA/device tree у AMD и BSP/linker regions/header files у Intel.

### 9. Какие layout-паттерны обычно плохи

**Случайное распределение адресов.**  
Когда блоки получают base address в порядке появления в проекте, без общей схемы.

**Чрезмерная “экономия” адресов.**  
Когда диапазоны нарезают вплотную и микроскопически, лишь бы не оставить пустых мест. Это почти всегда мешает расширению и делает карту нечитаемой.

**Смешивание разных классов ресурсов.**  
Например, рядом вперемешку DDR window, UART, DMA regs, custom CSR и debug scratchpad без видимой структуры.

**Отсутствие master-specific views.**  
Когда карту проектируют только “как видит CPU”, а потом оказывается, что DMA или accelerator должны видеть другой набор областей.

Все это не запрещено формально, но противоречит той системной модели, на которой построены адресные редакторы и interconnect generators у AMD/Intel.

### 10. Layout и software

На уровне SoC address space layout почти всегда становится частью software ABI платформы. У Intel BSP linker script опирается на available memory regions аппаратной системы, а generated BSP/header artifacts используют информацию из system design. У AMD device tree generator строит device tree по hardware configuration из XSA. Это означает простую вещь: layout нельзя считать “чисто аппаратной внутренней деталью”. Он почти неизбежно протекает в software.

Поэтому хороший layout должен быть:

- достаточно стабильным между ревизиями
- документированным
- согласованным между hardware design, BSP/device tree и драйверами

### 11. Практическая ментальная модель

Удобно думать так:

1. Сначала делишь всю карту адресов на **крупные районы**.
2. Потом внутри района задаешь **подрайоны** для подсистем.
3. Потом каждому slave даешь **base + range**.
4. Затем для каждого master проверяешь, **какие районы он реально видит**.
5. И только потом переходишь к offsets конкретных регистров внутри IP.

Так ты проектируешь именно **layout**, а не просто заполняешь address editor.

### 12. Что стоит запомнить

**Address space layout** — это архитектура всей карты адресов системы: как она разбита на крупные области, как выровнены диапазоны, как оставлен запас под рост и как разные master’ы видят разные сегменты. В FPGA/SoC это влияет не только на decode interconnect’а, но и на `fMAX`, масштабируемость дизайна и связность hardware/software контракта.
