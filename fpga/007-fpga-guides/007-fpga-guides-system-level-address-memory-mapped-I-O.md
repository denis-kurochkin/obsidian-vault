## Memory-mapped I/O

**Memory-mapped I/O (MMIO)** — это способ подключить периферию и пользовательские IP-блоки так, чтобы процессор или другой master обращался к ним **обычными чтениями и записями по адресам**, как будто это участок адресного пространства. В FPGA/SoC это основной способ управлять UART, timer, DMA, custom accelerator, debug bridge и регистровыми банками IP. В AMD/Xilinx мире для этого обычно используют **AXI4-Lite** или AXI memory-mapped интерфейсы, в Intel — **Avalon-MM**, а в Arm-системах низкоскоростная периферия часто сидит на **APB**, который прямо описывается как шина для timers, interrupt controllers, UARTs и I/O ports.

### 1. Что это значит на практике

Если у IP есть `base address`, то обращение по адресу `base + offset` попадает не в RAM-ячейку, а в **регистр устройства**. AMD в одном из reference design прямо показывает такую модель: аппаратные регистры memory-mapped в адресное пространство PS, а внутри блока используются **relative address** вроде `0x00`, `0x04`, `0x08` для control/status и interrupt-регистров.

То есть MMIO — это не “память для хранения данных”, а **программный вход в логику устройства**. Адрес здесь нужен не для доступа к массиву байтов, а для выбора нужного регистра: запустить блок, прочитать `idle/done`, разрешить interrupt, записать размер кадра, адрес буфера и так далее. AMD также прямо пишет, что AXI4-Lite IPIF используется как легковесный интерфейс между AXI interconnect и user IP, предоставляя address decoding по различным диапазонам адресов.

### 2. Чем MMIO отличается от обычной памяти

Главное отличие в том, что у обычной памяти смысл операций примерно такой: “прочитай или запиши значение по адресу”. У MMIO чтение и запись часто имеют **семантику управления устройством**. В приведенном AMD-примере один бит запускает обработку кадра, другие биты показывают `idle`, `ready`, `done`, а рядом расположены interrupt-регистры по фиксированным смещениям. То есть запись может **запустить действие**, а чтение — вернуть **текущее состояние блока**, а не просто “данные, лежащие в памяти”.

Поэтому MMIO обычно не используют для высокоплотной передачи потока данных, если речь не о специальных memory-mapped портах. Для bulk data в FPGA чаще применяют:

- **AXI4 memory-mapped** к BRAM/DDR, где важны burst’ы и throughput,
- или **AXI4-Stream**, который вообще не memory-mapped и предназначен для потока данных, а не для адресуемых регистров. AMD отдельно различает AXI4 как memory-mapped интерфейс, AXI4-Lite как light-weight single-transaction memory-mapped интерфейс, и AXI4-Stream как отдельный класс интерфейса.

### 3. Как MMIO выглядит в FPGA-системе

Типичная цепочка такая:

1. CPU, MicroBlaze, Nios, DMA или debug master выдает адрес и тип операции.
2. Interconnect делает **address decode**.
3. Выбирается нужный slave — например, регистры пользовательского IP.
4. Внутри IP адрес превращается в **offset** регистра.
5. Логика IP возвращает данные чтения или принимает данные записи.

И AMD, и Intel строят свои system-integration flow именно вокруг этого: memory-mapped master/slave интерфейсы соединяются через interconnect, а адрес решает, какой блок должен отвечать. Intel прямо описывает Avalon-MM master/slave как интерфейсы, через которые master и slave выполняют transfers через system interconnect.

### 4. Почему в FPGA чаще всего для MMIO используют AXI4-Lite или Avalon-MM

Потому что для регистров управления нужен не поток, а **простая адресуемая транзакция**. AMD определяет AXI4-Lite как **light-weight, single transaction memory-mapped interface**, а AXI4-Lite IPIF — как быстрый способ связать user IP с AXI interconnect и дать ему address decoding по диапазонам адресов.

В Intel-мире ту же роль обычно играет Avalon-MM: у периферии есть memory-mapped accessible registers, которые software или host master читает и пишет через адресуемый интерфейс. Даже в справочниках на периферийные IP разделы про регистры так и оформлены: “Avalon memory-mapped interface accessible registers”.

Практически это означает простую архитектурную развязку:

- **MMIO/AXI4-Lite/Avalon-MM** — для control/status,
- **AXI4 to memory** — для RAM/DDR и больших буферов,
- **AXI4-Stream** — для потока данных между блоками.

### 5. Типичная структура MMIO-регистров

У хорошо интегрированного IP обычно есть:

- `control` регистр,
- `status` регистр,
- `interrupt enable/status`,
- параметры конфигурации,
- иногда адреса/длины буферов для DMA или accelerator’а.

AMD в примере с Sobel filter показывает именно такую структуру: control/status по `0x00`, global interrupt enable по `0x04`, interrupt enable по `0x08`, interrupt status по `0x0C`, а доступ идет от заданного `base address`.

Это хороший паттерн для собственных IP, потому что software получает предсказуемую модель: сначала пишет конфигурацию, потом ставит `start`, потом читает `done/idle` или ждет interrupt. Тот же принцип лежит и в Intel peripheral IP, где разделы по software programming model и register map завязаны на memory-mapped доступные регистры.

### 6. Почему MMIO — это часть system-level architecture, а не только RTL одного блока

Потому что MMIO живет внутри **общей карты адресов SoC**. Устройство должно:

- получить `base address`,
- быть видимым master’ам,
- не конфликтовать диапазоном с другими slave,
- совпасть по адресам с software headers, BSP или device tree.

AMD documentation и Address Editor flow построены именно вокруг этого: address space master’а наполняется slave segments с назначенными адресами, а software потом опирается на этот hardware map.

Именно поэтому MMIO-недочеты часто выглядят как “логика вроде правильная, но драйвер читает не тот регистр”: проблема оказывается не в RTL регистра, а в base address, offset, visibility или неверно описанной карте адресов.

### 7. Где MMIO особенно уместен, а где нет

MMIO хорошо подходит для:

- конфигурации IP,
- статуса и диагностики,
- interrupt control,
- редких команд управления,
- настройки DMA/accelerator’а,
- доступа к небольшому числу CSR-регистров.

Он плохо подходит как основной путь для высокоскоростного payload-трафика, если ты начинаешь гонять через регистры большие массивы данных. Для этого AMD предлагает memory-mapped AXI4 с burst’ами, BRAM controller или доступ к DDR, а не AXI4-Lite регистры.

Это полезно запомнить как системное правило:  
**MMIO управляет datapath, но обычно не заменяет сам datapath**.  
То есть через MMIO ты говоришь блоку _что делать_, а сами данные чаще идут по stream или через memory-mapped RAM/DDR.

### 8. Типичные ошибки при проектировании MMIO

Самые частые ошибки такие.

**Смешать control path и data path.**  
Когда через MMIO пытаются передавать то, что должно было идти через FIFO, AXI4-Stream или память. Это почти всегда плохо по throughput и архитектуре.

**Сделать хаотичную карту регистров.**  
Когда offsets неструктурированы, отсутствуют reserved поля, непонятны semantics битов и сложно поддерживать software. В рабочих AMD/Intel примерах register map всегда оформляется как явная таблица base + relative offsets + meaning.

**Путать base address и offset.**  
Base задается на уровне системы, а конкретный IP обычно знает только внутренние offsets. AMD прямо показывает это разделение в документации на memory-mapped регистры.

**Не думать о software с самого начала.**  
Если регистры удобны только для RTL, но неудобны для драйвера, потом получается много ненужной glue-логики и ручных обходов. И AMD, и Intel документируют свои peripheral IP именно через software programming model / register map, что хорошо показывает правильный приоритет.

### 9. Практическая ментальная модель

Удобно думать так:

- **RAM**: “храни здесь данные”.
- **MMIO**: “управляй устройством через регистры”.
- **AXI4 memory-mapped**: “доступайся к большой адресуемой памяти и буферам”.
- **AXI4-Stream**: “гони поток данных без адресов”.

Если держать эту модель в голове, становится проще решать, какой интерфейс нужен новому IP: AXI4-Lite slave для control, AXI master для доступа к памяти, stream для datapath — или комбинация этих ролей.

### 10. Главное, что стоит запомнить

**Memory-mapped I/O** — это адресуемый доступ к регистрами устройства через системную шину. В FPGA/SoC это стандартный способ подключать периферию и custom IP к CPU, softcore и другим master’ам. Для MMIO характерны:

- `base address + offsets`,
- регистры control/status/interrupt,
- interconnect decode по адресу,
- разделение control path и bulk data path,
- тесная связь с общей address map платформы
