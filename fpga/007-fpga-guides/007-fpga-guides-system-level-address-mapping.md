## Address mapping в контексте FPGA System-Level / SoC Awareness

**Address mapping** — это правило, по которому системный адрес от master-блока превращается в доступ к конкретному slave-блоку.  
Иначе говоря, это ответ на вопрос:

**“Если CPU / DMA / accelerator выдал адрес `0x...`, какой именно блок системы должен на него ответить?”**

В контексте SoC это уже не тема отдельного IP, а тема **всей системы**: interconnect, мостов, памяти, периферии, debug-доступа и software. AMD Vivado описывает это через **master address space** и **slave segments**, которые назначаются в это пространство адресов; Intel Platform Designer показывает адресную карту по master-интерфейсам и base address для slave-интерфейсов.

---

### 1. Что именно здесь отображается

Обычно в системе есть:

- **master**: CPU, DMA, PCIe master, debug master, custom accelerator
- **slave**: BRAM controller, DDR controller, регистры IP, UART, timer, bridge, shared memory

У slave есть адресуемый диапазон:

- **base address** — начало диапазона
- **range / size** — размер диапазона
- **offset** — смещение внутри этого диапазона

Например:

- `0x4000_0000 ... 0x4000_FFFF` → регистры периферии A
- `0x8000_0000 ... 0x8FFF_FFFF` → DDR window
- `0xA000_0000 ... 0xA000_0FFF` → control/status регистры accelerator’а

Тогда чтение по адресу `0xA000_0010` означает:  
“иди в accelerator register block и возьми регистр по смещению `0x10`”.

---

### 2. Как это выглядит на уровне системы

Полезно разделять три понятия:

#### Address space master’а

Это то, **как систему видит конкретный источник транзакций**.  
У разных master’ов карта может быть разной. Vivado прямо говорит, что slave segment становится видимым **в адресном пространстве master’а** после назначения адреса. Intel в учебных материалах показывает, что один и тот же slave может иметь базовый адрес, видимый из разных master maps.

#### Slave segment

Это **адресуемая область**, которую предоставляет конкретный slave interface: память, регистровый банк, окно моста и т.д.

#### Address decode

Это логика interconnect’а, которая по входному адресу решает, **куда маршрутизировать транзакцию**. На AXI/Avalon это и есть основа memory-mapped interconnect.

---

### 3. Почему это важно именно для FPGA/SoC-интеграции

На уровне отдельного RTL-модуля address mapping может выглядеть простой темой: “у меня есть регистры по offset”.  
Но на уровне SoC появляются дополнительные вопросы:

- какие master’ы вообще видят этот блок
- не пересекаются ли диапазоны адресов
- не указывают ли два диапазона на один и тот же target по ошибке
- проходит ли доступ через мост в другой сегмент
- совпадают ли hardware map и software map
- корректно ли настроены base address, aperture, window, exclude/include

Vivado Address Editor даже различает состояния **Assigned**, **Unassigned** и **Excluded**. Это важно: блок может физически быть в дизайне, но быть **исключен из видимости конкретного master’а**.

---

### 4. Как реально проходит транзакция

Когда master делает memory-mapped доступ, система обычно проходит такую цепочку:

1. master выдает адрес и тип операции — read/write
2. interconnect сравнивает адрес с диапазонами slave’ов
3. выбирается target block
4. вычисляется локальный offset внутри target
5. транзакция проходит к slave или через bridge в другой сегмент
6. ответ возвращается обратно master’у

То есть address mapping — это не “табличка для документации”, а рабочая часть маршрутизации транзакций. На AMBA AXI это часть обычной memory-mapped коммуникации manager/subordinate; в инструментах AMD/Intel это отражается через editor/address map.

---

### 5. Один и тот же блок может быть виден по-разному

Это очень важная системная мысль.

Один и тот же slave не обязан одинаково выглядеть для всех master’ов.  
Например:

- CPU видит и DDR, и регистры периферии, и debug-блоки
- DMA видит только DDR и свой descriptor RAM
- custom accelerator видит только локальные буферы и control window
- debug master видит почти всё, но только через специальный путь

Именно поэтому address mapping — это не просто “список адресов”, а **отношение master → visible slave segments**. В AMD это прямо отражено через назначение slave segments в address space конкретного master’а.

---

### 6. Где чаще всего бывают ошибки

#### Overlap

Два slave получают пересекающиеся диапазоны.  
Тогда decode становится неоднозначным или инструмент запрещает такую конфигурацию.

#### Hole

В адресной карте есть “дырка”: адрес существует в пространстве master’а, но никуда не ведет.  
Такой доступ может дать decode error / bus error.

#### Alias

Два разных адреса ведут в один и тот же ресурс unintentionally.  
Иногда это делается специально, но чаще это ошибка маппинга или bridge window.

#### Wrong visibility

Нужный master не видит нужный slave, потому что segment не assigned или excluded.

#### HW/SW mismatch

В RTL и в software используются разные base address.  
В железе регистр по одному адресу, а драйвер обращается по другому.

Эти сценарии типичны настолько, что Vivado отдельно документирует addressing-related warnings и ошибки при validation block design.

---

### 7. Почему software зависит от address mapping напрямую

Address mapping — это контракт между hardware и software.

Если в системе есть CPU или firmware, то address map должен совпасть сразу в нескольких местах:

- block design / system integration tool
- RTL адресация внутри IP
- generated headers / BSP
- linker script
- device tree
- драйверы
- документация на регистры

Если здесь нет единого источника истины, появляются очень неприятные ошибки:

- “читается не тот регистр”
- “драйвер пишет в пустоту”
- “DMA не видит память”
- “CPU видит устройство, а accelerator нет”

Поэтому system-level awareness здесь означает: **думать об address map не как об internal detail, а как об интерфейсе всей платформы**.

---

### 8. Bridges, windows, remap

В более сложных системах адрес не всегда идет прямо в конечный slave.

Бывает, что он проходит через:

- **bridge** — переход между interconnect/domain/subsystem
- **address window** — только часть адресного пространства пробрасывается дальше
- **remap** — один и тот же адресный диапазон может быть перенаправлен по другому пути
- **span extender / aperture** — когда нужно связать master и target с разной видимой глубиной адресного пространства

AMD отдельно использует понятие **apertures** в addressing flow, а Intel Platform Designer и связанная документация рассматривают address windows/map across masters and slaves.

---

### 9. Самая полезная ментальная модель

Есть две разные, но часто путаемые темы:

#### Внутренняя адресация памяти

Например, в DDR: bank/row/column.  
Это **внутреннее устройство target’а**.

#### Системная address map

Например:

- `0x4000_0000` → UART
- `0x8000_0000` → DDR controller
- `0xA000_0000` → accelerator regs

Это **маршрутизация на уровне SoC**.

Когда говорят **address mapping** в контексте System-Level / SoC Awareness, почти всегда имеют в виду именно **второе**:  
как глобальное адресное пространство платформы распределено между системными ресурсами.

---

### 10. Практические правила для FPGA-разработчика

#### 1. Думай не только “где регистры IP”, но и “кто их видит”

Для одного master блок может быть доступен, для другого — нет.

#### 2. Планируй карту адресов заранее

Лучше заранее резервировать крупные непротиворечивые области:

- периферия
- control/status
- shared memory
- boot memory
- external memory windows

#### 3. Делай диапазоны понятными

Хорошо, когда по base address сразу видно класс устройства:

- периферия в одной зоне
- память в другой
- debug/service blocks отдельно

#### 4. Избегай случайного aliasing

Если alias нужен специально — документируй его явно.

#### 5. Держи один источник истины

Block design / system integration tool должен быть согласован с software headers и документацией.

#### 6. Проверяй карту адресов как часть integration verification

Нужно тестировать не только функциональность IP, но и:

- правильный decode
- доступность нужных slave’ов
- ошибки на несуществующие адреса
- visibility для разных master’ов

---

### 11. Главное, что стоит запомнить

**Address mapping на уровне SoC** — это назначение адресных диапазонов системным ресурсам и определение того, какой master по какому адресу какой slave видит. Это основа memory-mapped интеграции в FPGA-системах, а не просто служебная настройка редактора адресов.

Ключевые идеи:

- у master есть **address space**
- slave предоставляет **addressable segments**
- interconnect делает **decode + routing**
- у каждого сегмента есть **base address + range**
- разные master’ы могут видеть разные части системы
- address map должен совпадать между hardware и software
- ошибки address mapping — это типовые системные баги, а не мелкие детали интеграции
