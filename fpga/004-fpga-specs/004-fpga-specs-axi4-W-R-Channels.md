**AXI Write/Read Channels** — это разделение передачи данных в AXI на два независимых направления:

- **Write path (запись)**
- **Read path (чтение)**

Каждое направление состоит из отдельных каналов, которые работают независимо и используют собственный handshake.

Проще говоря, это ответ на вопрос:

> как в AXI организованы операции чтения и записи и как они разбиты на каналы?

Главная особенность:

> чтение и запись в AXI полностью декомпозированы и не блокируют друг друга

---

## Главная идея

AXI разделяет операции на:

- write transaction
- read transaction

И реализует их через разные каналы.

Главная практическая мысль:

> в AXI нет "одного канала данных" — есть независимые каналы адреса, данных и ответа

Это позволяет:

- pipeline
- параллелизм
- high throughput

---

## Write Channels (каналы записи)

Write path состоит из трёх каналов:

- AW — write address
- W — write data
- B — write response

---

### 1. AW (Write Address Channel)

Передаёт:

- адрес записи
- параметры burst

Сигналы:

- AWADDR
- AWLEN
- AWSIZE
- AWBURST
- AWVALID / AWREADY

Главная роль:

> описывает "что и куда писать"

---

### 2. W (Write Data Channel)

Передаёт:

- данные

Сигналы:

- WDATA
- WSTRB
- WLAST
- WVALID / WREADY

Главная роль:

> передаёт сами данные (один или несколько beats)

---

### 3. B (Write Response Channel)

Передаёт:

- результат операции

Сигналы:

- BRESP
- BVALID / BREADY

Главная роль:

> подтверждает завершение записи

---

## Как работает write transaction

Логически:

1. AW — адрес
2. W — данные
3. B — ответ

Но важно:

- AW и W независимы
- данные могут идти позже или раньше адреса

Главная идея:

> write transaction не является "атомарной" — она разбита на части

---

## Read Channels (каналы чтения)

Read path состоит из двух каналов:

- AR — read address
- R — read data

---

### 1. AR (Read Address Channel)

Передаёт:

- адрес чтения
- параметры burst

Сигналы:

- ARADDR
- ARLEN
- ARSIZE
- ARBURST
- ARVALID / ARREADY

Главная роль:

> описывает "что читать"

---

### 2. R (Read Data Channel)

Передаёт:

- данные
- статус

Сигналы:

- RDATA
- RRESP
- RLAST
- RVALID / RREADY

Главная роль:

> возвращает данные по одному beat

---

## Как работает read transaction

Логически:

1. AR — запрос
2. R — поток данных

Важно:

- данные могут приходить с задержкой
- приходят по одному beat
- последний beat помечается RLAST

---

## Независимость каналов

Ключевая особенность AXI:

- AW, W, B, AR, R работают независимо
- каждый имеет свой VALID/READY

Это означает:

- адрес может прийти раньше данных
- данные могут задерживаться
- read и write не блокируют друг друга

---

## Write vs Read — ключевые различия

Write:

- три канала
- есть отдельный ответ (B)
- данные идут от master к slave

Read:

- два канала
- ответ встроен в R
- данные идут от slave к master

---

## Handshake в каналах

Каждый канал использует:

- VALID
- READY

Передача происходит при:

VALID && READY

Важно:

- каждый канал управляется отдельно
- нет глобальной синхронизации

---

## Связь между каналами

Хотя каналы независимы, есть логическая связь.

Write:

- AW задаёт параметры
- W передаёт данные
- WLAST завершает burst
- B подтверждает

Read:

- AR задаёт параметры
- R передаёт данные
- RLAST завершает burst

---

## Pipeline и параллелизм

AXI позволяет:

- отправлять новый AW, пока предыдущий W ещё идёт
- иметь несколько AR "в полёте"
- получать данные позже

Это даёт:

- latency hiding
- высокую загрузку интерфейса

---

## Частые ошибки

Ошибка 1  
Связывать AW и W напрямую

---

Ошибка 2  
Ожидать, что данные придут сразу после адреса

---

Ошибка 3  
Игнорировать независимость каналов

---

Ошибка 4  
Неправильно обрабатывать B channel

---

Ошибка 5  
Забывать про RLAST / WLAST

---

## Debug мышление

Если что-то не работает, проверять:

- есть ли handshake на каждом канале
- не завис ли READY
- не потерян ли LAST
- приходит ли B response
- есть ли задержка между AR и R

---

## Практические рекомендации

1. Буферизовать каналы независимо
2. Не связывать READY разных каналов
3. Учитывать задержки между каналами
4. Использовать pipeline
5. Проверять LAST

---

## Пример мышления

Вместо:

- "я делаю запись"

думать:

- когда ушёл AW
- когда пошёл W
- когда пришёл B

И отдельно для чтения:

- когда ушёл AR
- когда начали приходить R

---

## Главное, что нужно запомнить

AXI делит операции на независимые каналы:

- write: AW, W, B
- read: AR, R

Главный practical смысл:

- decoupling
- pipeline
- параллелизм

Золотое правило:

> думай о каждом канале отдельно, даже если логически это одна операция

---

## Краткая памятка

Write channels  
AW — адрес  
W — данные  
B — ответ

Read channels  
AR — адрес  
R — данные

Что важно помнить  
каналы независимы  
у каждого свой handshake  
данные и адрес не синхронны

Золотое правило  
AXI — это не "шина", а набор независимых потоков

---

## Мини-шаблон инженерного анализа

Для любой транзакции полезно ответить:

1. Когда идёт адрес?
2. Когда идут данные?
3. Когда приходит ответ?
4. Есть ли задержки между каналами?
5. Есть ли backpressure?
6. Где может быть stall?
7. Все ли каналы корректно работают?

Если это понятно — AXI система будет предсказуемой и устойчивой.
