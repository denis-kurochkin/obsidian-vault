**AXI handshake (VALID / READY)** — это механизм согласования передачи данных между двумя блоками (producer и consumer), который определяет **точный момент передачи** в каждом канале AXI.

Он используется во всех каналах:

- AW, W, B
- AR, R
- а также в AXI-Stream

Проще говоря, это ответ на вопрос:

> в какой момент данные действительно переходят от одного блока к другому?

Важно понимать: наличие данных на шине **не означает**, что они переданы.

Передача происходит только при выполнении условия:

- VALID = 1
- READY = 1

---

## Интуитивная идея

Представь два блока:

- один хочет передать данные;
- второй хочет их принять.

Первый говорит:

- "у меня есть данные" → VALID = 1

Второй говорит:

- "я готов принять" → READY = 1

Передача происходит только тогда, когда оба сигнала равны 1 **в один и тот же такт**.

Если хотя бы один из сигналов = 0 → передачи нет.

Это не push и не pull модель — это **двустороннее согласование**.

---

## Главная идея

AXI handshake определяет, что:

> данные считаются переданными только в момент VALID && READY

Каждое такое событие — это один **beat передачи**.

Главная практическая мысль:

> данные "существуют" для системы только в момент handshake, а не тогда, когда они просто выставлены на шине

---

## Роли сторон

### Producer (источник)

- управляет VALID
- выставляет данные (ADDR, DATA, CONTROL)

### Consumer (приёмник)

- управляет READY
- принимает данные

Важно:

- VALID и READY генерируются **независимо**
- ни одна сторона не должна "ждать" другую

---

## Ключевые правила AXI

### 1. VALID не зависит от READY

Producer не должен писать:

- "если READY = 1, тогда выставлю VALID"

Это приводит к deadlock, если consumer делает то же самое.

Правильная модель:

- VALID определяется только внутренним состоянием producer

---

### 2. Данные должны быть стабильны

Если:

- VALID = 1
- READY = 0

то данные уже "предложены" к передаче.

Поэтому:

- они должны оставаться неизменными
- до момента handshake

Иначе возникает data corruption.

---

### 3. Один handshake = один beat

Каждое совпадение VALID и READY:

- это ровно одна передача

Burst — это просто последовательность таких событий.

---

### 4. READY может быть 0 в любой момент

Consumer может в любой такт сказать:

- "я не готов" → READY = 0

Это нормальный механизм — **backpressure**.

---

## Timing сценарии

### Идеальный случай

- VALID = 1 постоянно
- READY = 1 постоянно

→ передача каждый такт  
→ максимальный throughput

---

### Backpressure

- READY периодически становится 0

→ данные удерживаются  
→ передача останавливается

---

### Late READY

- VALID уже = 1
- READY долго = 0

→ данные "висят"  
→ потом передаются

---

### Burst с паузами

- VALID = 1
- READY скачет

→ burst растягивается во времени

---

## Backpressure

Backpressure — это управление потоком через READY.

Причины:

- FIFO заполнен
- downstream занят
- arbitration
- ограничение ресурсов
- CDC

Влияние:

- увеличивает latency
- снижает throughput
- может распространяться по системе

Важно:

backpressure — это **не ошибка**, а часть архитектуры.

---

## Pipeline и throughput

AXI — fully pipelined интерфейс.

Если нет backpressure:

- передача = 1 beat / такт

Это достигается за счёт:

- независимости каналов
- локального handshake
- отсутствия глобальных блокировок

---

## Частые ошибки

### Ошибка 1. VALID зависит от READY

→ deadlock

---

### Ошибка 2. READY комбинаторно зависит от VALID

→ combinational loop  
→ плохой timing

---

### Ошибка 3. Данные меняются при VALID=1 и READY=0

→ потеря данных

---

### Ошибка 4. Неправильный LAST

→ ломает burst  
→ критично для DMA / DDR

---

### Ошибка 5. Связывание каналов

→ нарушает независимость AXI

---

## Handshake и timing closure

READY часто становится критическим сигналом.

Почему:

- проходит через FIFO
- проходит через interconnect
- зависит от сложной логики

→ длинный combinational path

---

## Как это решается

- pipeline READY
- register slice
- skid buffer
- разбиение логики

---

## Skid buffer

Skid buffer — это небольшой буфер (обычно 1–2 элемента), который:

- разрывает длинный путь READY
- принимает данные при временном stall
- сохраняет throughput

Это стандартный элемент production RTL.

---

## AXI4 vs AXI-Stream (в контексте handshake)

Handshake одинаковый, но смысл разный.

AXI4:

- привязан к адресам
- важен порядок

AXI-Stream:

- поток данных
- backpressure влияет на всю pipeline

---

## Handshake и CDC

Handshake сам по себе **не решает** clock domain crossing.

Нужно использовать:

- async FIFO
- synchronizers

Ошибка:

- напрямую соединять VALID/READY между доменами

---

## Debug (ILA)

Что смотреть:

- VALID=1, READY=0 → stall
- READY=1, VALID=0 → idle
- нет LAST → ошибка burst
- нерегулярный поток → backpressure

Типичные симптомы:

- READY всегда 0 → блокировка
- низкий throughput → частые stalls

---

## Практическое мышление

Вместо:

- "AXI не работает"

нужно спрашивать:

- кто producer, кто consumer
- как формируется VALID
- как формируется READY
- где возникает stall
- есть ли буферизация
- есть ли длинный путь READY

---

## Главное, что нужно запомнить

AXI handshake — это механизм согласования передачи через VALID и READY.

Передача происходит только при VALID && READY  
Данные должны быть стабильны до передачи  
READY управляет backpressure  
Каждый handshake — один beat

Главный practical смысл:

правильная реализация handshake определяет  
и корректность, и производительность всей AXI системы.

---

## Краткая памятка

### Что такое handshake

Согласование передачи через VALID/READY

### Когда происходит передача

Только при VALID && READY

### Главные правила

- VALID не зависит от READY
- данные стабильны при VALID=1 и READY=0
- READY может быть 0

### Что важно помнить

- backpressure — нормален
- READY — часто критический путь
- pipeline обязателен

### Золотое правило

если handshake реализован неправильно — AXI система работать корректно не будет, даже если всё остальное написано правильно

---

## Мини-шаблон инженерного анализа

Для любого AXI интерфейса полезно ответить:

1. Кто producer и consumer?
2. Кто генерирует VALID и READY?
3. Где возможен backpressure?
4. Есть ли буферизация?
5. Удерживаются ли данные корректно?
6. Есть ли длинный combinational путь?
7. Есть ли риск deadlock?

Если эти вопросы закрыты — handshake реализован правильно и система будет стабильной и масштабируемой.
